# AI Workflow — エージェント連携による開発プロセス

> 位置づけ: 3 つの専門エージェント（`dev-architect` / `biblical-editor` / `product-growth`）を、
> **CLAUDE.md（憲法）の原則に従った単一の開発プロセス**として連携させる運用フロー。
> エージェント定義は `.claude/agents/`。上位規範は常に `CLAUDE.md`。
> 本書はプロセスを定めるものであり、憲法を上書きしない。矛盾が生じたら CLAUDE.md が勝つ。

---

## 0. 大原則（この文書の骨格）

1. **エージェントは独立して動かない。** 提案は必ず 3 つの視点（価値・忠実性・技術）を通過してから実装される。
   曖昧・広範な依頼（「直して」「対応して」「全部やって」）は、まず product-growth が **Value Brief へ変換**してから G1 を開始する（非構造の依頼を直接実装しない）。
2. **各ゲートは「通す/差し戻す」の二択。** 差し戻しは失敗ではなく設計の一部（Unresolved by Design と同じ思想）。
3. **判断の最上位は常に CLAUDE.md §1 First Principle。** どのエージェントも原則を緩める権限を持たない。
4. **既存の「設計 → 実装 → 監査 → 凍結」（CLAUDE.md §9）に重ねる。** 本フローはそれを置き換えず、各段に「誰が何を見るか」を割り当てる。

```
[提案] → G1 価値ゲート → G2 忠実性ゲート → G3 技術ゲート
        (product-growth)  (biblical-editor)   (dev-architect)
                                                    │
                                              [実装前レビュー]
                                                    │
                                                 [実装]
                                                    │
                                              [実装後監査] → [凍結/記録]
                                        (dev-architect + biblical-editor)
```

> G2 の判定が技術的事実に依存するときは、G2⇄G3 の **Evidence Request**（§2）で証拠を取得してから G2 を確定する。
> 順序（価値→忠実性→技術）は保つが、忠実性判定に必要な証拠だけを先に取りに行く。

## 1. 責務境界（三すくみ）

| エージェント | 問い | 守るもの | 権限外（引き継ぐ先） |
|---|---|---|---|
| **product-growth** | それは読む人の何を良くするか？ | 利用者価値・優先順位・伝え方（§1/§2） | 忠実性 → biblical-editor / 技術 → dev-architect |
| **biblical-editor** | それは原語構造に忠実か？ | L-0 境界・Reading Japanese の声（§1 非矛盾/§3） | 価値 → product-growth / 技術 → dev-architect |
| **dev-architect** | それは安全に作れるか？ | アーキテクチャ・FROZEN・regression（§4/§9/§10） | 内容 → biblical-editor / 価値 → product-growth |

**境界の鉄則:** どのエージェントも自分の権限外の理由で他の判断を上書きしない。
たとえば dev-architect は「実装が楽だから」で語義文面を変えない。product-growth は「価値が高いから」で L-0 を緩めない。

**定義と検証の分離:** 判断が「意味論の定義」と「engine 挙動の検証」の両方に跨るとき、両者を別責務として切り分ける。
たとえば「何を決定的構造（固定点）と呼ぶか」の**定義**は biblical-editor が持ち（意味論的判断）、
「ある語がその定義を満たすか」の**検証**は dev-architect が持つ（engine 再処理などの機械判定）。
忠実性の合否が技術的事実に依存する場合は、§2 の Evidence Request でこの定義と検証を橋渡しする。
これは新しい権限の付与ではなく、既存の三すくみ責務を跨ぐ判断の割り当てを明示するもの。

## 2. 機能提案から実装までの連携順序

### G1. 価値ゲート — product-growth（「作る意味があるか」）

- **入力:** 機能提案（誰の・どの読書体験を・どう良くするか）。
- **判定:** §1 First Principle と §2 Mission に照らす。
  - 「原著者の意図理解＝読みにどう貢献するか」を言語化できるか。
  - 「技術的に面白い/可能」だけを根拠にしていないか。
  - 読書が主・研究が従の序列を崩さないか。
  - **品質・基盤改善（新機能でない）の場合**は、価値を「除去する読書摩擦」として言語化する。摩擦除去も §1 貢献の正規な形であり、価値が拡散的だからと不当に落とさない（逆に、消す摩擦を特定できないものは通さない）。
- **通過条件:** 貢献が読書体験の言葉で説明できる。
- **差し戻し:** 貢献を説明できない → **却下または保留**（優先度を下げる）。
- **出力（次へ渡す情報）→ §4 の Value Brief。**

### G2. 忠実性ゲート — biblical-editor（「原語を裏切らないか」）

- **入力:** Value Brief。特に「読者に何をどう見せるか（文面・グロス・ヒント）」。
- **判定:** L-0 境界（§3）に照らす。
  - この提案は実現時に **翻訳/推論/語義選択/自然化/未判定補完**を要求しないか。
  - 「読者のため訳した方が親切」という誘惑を含んでいないか（§1 非矛盾）。
  - 露出禁止物（分類ラベル・confidence 数値・原語・語形コード）を読む面に出さずに実現できるか。
- **通過条件:** L-0 を侵さずに実現できる設計余地がある。
- **差し戻し:** L-0 を侵す前提でしか成立しない → **却下**、または「解釈を足さない忠実版」への再設計を要求。
- **Evidence Request（G2⇄G3 の証拠依頼）:** L-0 判定が技術的事実に依存する場合（例: 「engine 再処理後も値が変わらない固定点か」）、biblical-editor は判定を保留し dev-architect へ **Evidence Request** を出す。dev-architect は機械判定の結果（証拠）だけを返し、内容の是非判断はしない（定義と検証の分離・§1）。biblical-editor はその証拠を根拠に Fidelity Note を確定する。これは順序の例外ではなく忠実性判定に必要な証拠取得であり、G3 の技術ゲート本審査（影響分析・回帰設計）を前倒ししない。
- **出力 → Value Brief に Fidelity Note を付す（§4）。**

### G3. 技術ゲート — dev-architect（「安全に作れるか」）

- **入力:** Value Brief + Fidelity Note。
- **判定:** §4/§9/§10 に照らす。
  - 触れる層は？ FROZEN層（reading-engine Phase1–7 / reading-context Stage D / presentation-policy / phrase-renderer / morph-rule-engine-v1）に触れるか。
  - ディレクトリ責務・旧パス禁止・SSOT（ResolveContext / roadmap.json / tokens.css）を侵さないか。
  - 影響範囲は何トークン・どの表示・どの層へ波及するか。
  - **既存資産との重複はないか。** 既存の反映実績・台帳・FROZEN 済み範囲と差分を取り、「何が未対応で、なぜ今か」を確認する。二重作業・既存資産の破壊を防ぐ。
- **通過条件:** アーキテクチャ境界を侵さず、影響範囲と検証手段が見積もれる。
- **差し戻し:** 境界侵害・影響不明・回帰不能 → **設計に戻す**。
- **出力 → Change Plan（§4）。ここで初めて実装に進む。**

> **順序の理由:** 価値 → 忠実性 → 技術の順にするのは、
> 「作る価値のないもの・原語を裏切るもの」を**実装コストを払う前に落とす**ため。
> 技術ゲートを最後に置くのは、最も高コストな検討（影響分析・回帰設計）を、
> 前二門を通った提案にだけ払うため。

## 3. 実装前レビュー / 実装 / 実装後監査

### 3.1 実装前レビュー（dev-architect 主宰）

Change Plan に対し、実装着手前に確定する：

- **QA 設計:** どの回帰（`npm run test:re-*`）・どの監査（例 `node scripts/concept-audit.cjs`）・どの基準値で検証するか。
- **regression 防止:** FROZEN層・コアに触れるなら、対応する回帰テストへのケース追加を**着手条件**とする（テスト無しの着手を認めない）。
- **rollback 準備:** 「機械適用 + 三重 rollback（diff / git / 台帳）」の経路を確認（§9）。
- **before スナップショット:** 変更前の基準値（トークン数・一致率・表示差分）を記録。

### 3.2 実装

- Change Plan と実装前レビューの合意範囲のみを変更する。範囲外へ広げない。
- `core/` は DOM/window 非依存を保つ。`assets/data/` は `scripts/` 経由でのみ生成。
- 実装はエージェントの分担に従う：**技術実装 = dev-architect**。文面・グロス・ヒントの**内容は生成せず**、biblical-editor が承認した文言・方針の範囲に留める。

### 3.3 実装後監査（dev-architect + biblical-editor の二重監査）

| 監査 | 担当 | 合格条件 |
|---|---|---|
| **回帰監査** | dev-architect | `npm run test:re-*` 全 PASS。FROZEN 基準値を割らない。表示差分が想定内。 |
| **影響監査** | dev-architect | 波及トークン・層が事前見積もりと一致。想定外の波及ゼロ。 |
| **忠実性監査** | biblical-editor | 出力文面が L-0 を侵さない（翻訳/推論/自然化/露出なし）。原語構造の過不足なし。 |
| **概念/データ整合** | dev-architect | 概念データ変更時は `concept-audit.cjs` が FAIL しない。 |

- いずれか不合格 → **rollback（三重経路）して設計に戻す**。「とりあえず出す」をしない。
- product-growth は最後に**利用者視点レビュー**（読みの流れが途切れないか・伝え方が Mission のトーンか）を行うが、これは技術/忠実性監査を代替しない。

### 3.4 凍結・記録

- 完成した層は CLAUDE.md §10 の FROZEN プロトコルに従い、ヘッダへ `[FROZEN 日付]` と基準値を記録。
- ロードマップに関わる変更は `assets/data/roadmap.json`（単一ソース）のみ更新。
- Editorial Asset（`reading-hints.json` 等）の変更は台帳・version/status を更新（bible_data と分離）。

## 4. エージェント間で渡す情報（Handoff Artifacts）

情報は「口頭の要約」ではなく、次のゲートが判定に使える**構造化されたメモ**として渡す。
（実体は会話上のテキストで可。フォーマットを揃えることが目的。）

### Value Brief（product-growth → biblical-editor / dev-architect）
```
提案:            （何を）
種別:            新機能 / 品質・基盤改善
対象読者:         （誰の読書体験か）
貢献:            （原著者の意図理解＝読みへの貢献を1〜2文で）
除去する読書摩擦:  （種別＝品質・基盤改善のとき必須: どの引っかかりを消すか。
                 摩擦除去も §1 貢献の正規な形。新機能のときは任意）
非目標:           （これは何をしないか。研究ツール化しない等）
優先度理由:       （§1 に照らした根拠）
```

### Fidelity Note（biblical-editor が付す）
```
L-0 判定:     OK / 条件付きOK / NG
触れる境界:   （翻訳/推論/語義選択/自然化/未判定補完のどれに近いか）
露出リスク:   （ラベル/数値/原語/語形コードが読む面に漏れないか）
忠実版の条件: （足さずに実現するための制約。NG時は再設計要求）
```

### Change Plan（dev-architect が出す）
```
触れる層:     （core のどれ / index.html / assets/data など）
FROZEN接触:   有 / 無（有なら対象層と基準値）
影響見積り:   （波及トークン数・表示差分・層）
検証設計:     （実行する回帰・監査・比較方法）
rollback:    （diff/git/台帳の経路）
```

### Audit Report（実装後・dev-architect + biblical-editor）

Validation Criteria 形式。各項目に**合格条件と実測**を並べ、**全項目 PASS で凍結へ**、
いずれか FAIL で **rollback（三重経路）して設計へ**戻す（§3.3 と同じ監査を、判定可能な表として渡す様式）。

```
| 検証項目   | 担当           | 合格条件                                                  | 実測 | 判定      |
|-----------|---------------|----------------------------------------------------------|-----|-----------|
| 回帰      | dev-architect  | npm run test:re-* 全 PASS                                 | …   | PASS/FAIL |
| 基準値    | dev-architect  | FROZEN 基準値を割らない                                    | …   | PASS/FAIL |
| 影響範囲  | dev-architect  | 波及が事前見積りと一致・想定外0                              | …   | PASS/FAIL |
| 忠実性    | biblical-editor| L-0 非侵害（翻訳/推論/自然化/露出なし）・原語構造の過不足なし   | …   | PASS/FAIL |
| 品質指標  | 提案ゲート担当   | before/after で目標指標が改善または不変（例 語彙一貫性 M-16 の REVIEW 欠陥数） | … | PASS/FAIL |
| 利用者視点| product-growth | 読みの流れが途切れない・Mission のトーン                     | …   | PASS/FAIL |

総合判定:   全項目 PASS → 凍結・記録（§3.4）へ ／ FAIL → rollback して設計へ
```

## 5. 例外とエスカレーション

- **3 エージェントの判断が衝突したら** → CLAUDE.md §1 First Principle が裁定基準。「読み（原著者の意図理解）に最も忠実な選択」を採る。それでも決まらなければ、**保留（Unresolved by Design）**にして人間の判断を仰ぐ。決め打ちで埋めない。
- **未解決の構成不整合**（`pages/` vs `public/` のパス問題・CLAUDE.md §9/§10 記載）に触れる作業は、dev-architect が着手前に**人間へ確認**する。
- **憲法（CLAUDE.md）自体の改定が必要と判断された場合** → 本フロー内で勝手に変えない。人間の承認を得て CLAUDE.md を更新し、その後に本書・各エージェント定義を追随させる。

---

## 6. まとめ（一目で）

- **価値（product-growth） → 忠実性（biblical-editor） → 技術（dev-architect）** の順で門を通す。
- 忠実性が技術的事実に依存するときは **Evidence Request（§2）** で証拠を取り、**定義（editor）と検証（architect）を分離**する。
- 実装前に **QA・回帰・rollback** を確定し、実装後に **Validation Criteria 形式の二重監査**をする。
- 情報は **Value Brief（種別つき）/ Fidelity Note / Change Plan / Audit Report** で渡す。
- 迷ったら **CLAUDE.md §1**。埋めずに保留する勇気を持つ。
