## 3. UX 影響監査

### 3-1. 技術的確認

**分布パネル（dist-panel）— 最大の問題点:**

`_renderBookDist()` はトークンの `tok.bookKey` を集計し、頻度降順でソート。NT/LXX の区別なし。  
κύριος を検索すると分布パネルに「詩篇818・エレミヤ607・申命記558…」が表示される。NT の分布は **可視領域から外れる**（LXX が top 12 を独占）。

**件数表示:**

`result-count` に「8,172 件」と表示されるが、NT 714 / LXX 7,458 の内訳は示されない。

**Hit card:**

- `data-corpus` 属性なし
- NT/LXX の識別バッジなし
- LXX トークンは `japanese: null`（日本語グロスなし）
- 書物参照は日本語書名（「詩篇2章2節」など）で表示されるが、「LXX」の明示なし

**ソート順:**

NT 正典順（MAT → MRK → ... → REV）が先、LXX 正典順（GEN → EXO → ... → MAL）が後。  
page 1 は NT 結果のみ表示されることが多いが、分布パネルは全 corpus の集計を表示する。

### 3-2. UX 評価

| クエリ | 評価 | 理由 |
|--------|------|------|
| χριστός | **GOOD** | NT 固有用語。LXX 出現は稀 |
| κύριος | **BLOCK** | 8,172件の91%がLXX。分布に詩篇・エレミヤが先頭表示 |
| πνεῦμα | **IMPROVE** | NT/LXX がほぼ半々。LXX の旧約霊概念は補完価値あり |
| σωτηρία | **IMPROVE** | 70% LXX。メシア預言文脈で価値あり。件数表示は要注意 |
| διαθήκη | **BLOCK** | 90% LXX。NT文脈（契約の盃）を探すと旧約契約が圧倒 |

**評価基準:**
- **GOOD**: LXX 混在が教育的価値を持ち、説明不要
- **IMPROVE**: 分離表示で価値が増す
- **BLOCK**: 初心者を混乱させる（想定した検索結果と大きく異なる）

## 4. 表示方針 3 案の比較

### 案 A: NT のみ表示

| 観点 | 内容 |
|------|------|
| 方法 | `unifiedSearch` に corpus フィルタ追加。LXX 結果を除外 |
| メリット | 「新約聖書検索」として明快。初心者向き。件数が現実的 |
| デメリット | 既存 LXX データ（77.7% = 479,623 tokens）の価値を失う。κύριος が 8,172件 → 714件 |
| リスク | データ資産の損失。既存ユーザーの期待変化 |

### 案 B: 全表示 + ラベル分離

| 観点 | 内容 |
|------|------|
| 方法 | Hit card に `data-corpus` / NT/LXX バッジ追加。分布パネルを NT/LXX に分割表示。件数を「NT 714件 / LXX 7,458件」と表示 |
| メリット | 研究価値維持。NT 学習者も混乱しない |
| デメリット | UI 実装量増（バッジ・件数表示・分布パネル分割）。デザイン複雑化 |
| リスク | 分布パネル分割は大きな UI 変更を伴う |

### 案 C: デフォルト NT、詳細で LXX 表示（推奨）

| 観点 | 内容 |
|------|------|
| 方法 | デフォルト検索は NT のみ。「LXX を含む」チェックボックスまたはトグルで拡張 |
| メリット | 初心者と研究者を両立。既存アーキテクチャと整合（NT-first loading 実装済み） |
| デメリット | 実装コスト高（フィルタ UI + 検索ゲート修正） |
| リスク | LXX が見えにくくなる（研究者には不便） |

### 推奨: **案 C**

理由:
1. 既存アーキテクチャが「NT 先行ロード（`IDX.ready`）→ LXX バックグラウンド（`IDX.l3Ready`）」を実装済み。この設計意図が「NT が主、LXX が補足」を示唆する
2. 案 A（NT 限定）は LXX 資産を破棄する。投資コストが無駄になる
3. 案 B（全表示ラベル分離）は研究者向けだが、初心者体験を犠牲にする
4. 案 C（NT デフォルト）は UI 追加のみで既存データを活かせる

**最小実装（案 C の MVP）:**

優先度 HIGH：
- 結果件数に corpus 内訳を追加: 「NT 714件 ＋ LXX 7,458件（計 8,172件）」
- 分布パネルに corpus セクション境界線を追加（最低限の視覚分離）

優先度 MEDIUM：
- 「NT のみ」フィルタトグル追加

優先度 LOW：
- Hit card に NT/LXX バッジ付与

## 5. Design System への追加提案

以下を `docs/search-design-system.md` の新セクションとして追加することを提案する。

### 追加案: §11 Corpus Visibility Rule（コーパス可視性原則）

```markdown
§11 Corpus Visibility Rules

11-1. 複数 corpus が検索対象の場合、結果ヘッダーに corpus 内訳を明示する。
      例: 「NT 714件 / LXX 7,458件（計 8,172件）」

11-2. 分布パネル（dist-panel）は corpus ごとに区画を分けるか、
      corpus バッジを各書物バーに付与する。

11-3. Hit card には corpus を示す識別子を付与する。
      最低限: data-corpus="NT" / data-corpus="LXX" 属性。
      推奨: ユーザーに見える NT / LXX バッジ。

11-4. アプリのデフォルト検索 corpus を設計上明確に定める。
      この app の一次 corpus は NT（新約聖書）である。

11-5. LXX は「比較参照コーパス」として位置づける。
      NT 文脈で完結しない場合の補足情報として提示する。
```
