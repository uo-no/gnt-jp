# Phase UX-8.5 — Search Architecture Audit

**生成日**: 2026-07-14  
**対象**: search-tool.html / index.html / StudyPanel / 語順フロー / Concept Insight  
**方針**: 監査のみ。実装変更なし。

---

## 1. 現在のユーザーフロー

### Flow A — 探す → 読む → 理解する

```
search-tool.html
  └─ 検索（日本語 / ギリシャ語 / Strong番号）
       └─ Hit card（検索結果）
            └─ 「本文で見る」（新規タブ）
                 └─ index.html（章表示）
                      └─ 節をタップ
                           └─ StudyPanel Level 2（ことばの流れ / 語順フロー）
                                └─ chip をタップ → ドロワー
                                     └─ 「詳しく見る」
                                          └─ StudyPanel Level 3（単語詳細）
```

**評価**: 各画面の責務が明確に連鎖している。「探す → 読む → 理解する」の3段が自然な深さで接続されている。

---

### Flow B — 探す → 語順プレビュー（検索ページ内）

```
search-tool.html
  └─ Hit card
       └─ 「原文を見る」
            └─ hit-snippet-body が展開
                 └─ wlv-flow-stream（語順チップ）
                      ※ 表示のみ。クリック不可。
```

**評価**: ⚠️ 問題あり。

- index.html の語順フロー（ことばの流れ）と **同じ見た目のチップ** が展開するが、index.html ではチップをクリックすると単語詳細が開く（インタラクティブ）のに対し、search-tool.html では **クリック不可**（表示専用）
- 同一の UI 要素（`.wlv-chip`）が画面によって挙動が異なる
- 「原文を見る」というラベルは **ギリシャ語テキストを見せる** と読めるが、実際に表示されるのは **日本語グロスをギリシャ語順に並べたもの**

→ ラベルと実態がずれている。UI の一貫性も欠く。

---

### Flow C — 探す → テーマを知る → 探す（循環）

```
search-tool.html
  └─ Concept Insight（概念カード）
       ├─ 関連するギリシャ語 → 別語の検索へ
       ├─ 関連する概念 → 別テーマの検索へ
       └─ さらに詳しく → パターン検索 / 構文分類
```

**評価**: GOOD。テーマの「網」をたどる循環探索が機能している。ただし Concept Insight と通常の検索結果の関係（「これは検索結果の一部か、別情報源か」）が初見では不明確。

---

## 2. 各画面の役割定義

### search-tool.html（検索ツール）

**何をする場所か**:
- 語・テーマ・番号で聖書全文を **探す**
- 探した結果を **発見する**（どの書物に何件あるか、どんな文脈か）
- テーマを **知る**（Concept Insight による入口）

**何をしない場所か**:
- 本文を **読む**（読書行為は index.html に委譲）
- 語を深く **理解する**（語の詳細は StudyPanel に委譲）
- 訳を **比較する**（並列表示は index.html の機能）

**現状の逸脱点**:
- Hit card に日本語翻訳が自動表示される → **「読む」要素が検索ページに侵入** している
- "原文を見る" による語順チップ展開 → **「分析の入口」を中途半端に提供** している。インタラクティブでないため、中途半端な深さで止まる

---

### index.html（本文ページ）

**何をする場所か**:
- 聖書本文（ギリシャ語 + 翻訳）を **読む**
- 節単位で **選択・ナビゲート** する
- StudyPanel を起動して語を **探る**（入口）

**核心機能**: 「文書としての聖書」に向き合う場所

---

### 語順フロー（ことばの流れ / wlv-flow-stream in index.html）

**何をする場所か**:
- ギリシャ語の語順を保持した日本語グロスの列を **見る**
- 語を **選択する**（チップタップ → ドロワー / 詳しく見る）
- 「読む」から「理解する」への **橋渡し** を担う

**設計思想** (コードコメント `8534`):
> 流れ（デフォルト） … ギリシャ語順の日本語chip列。「読解UI」のメイン。

**位置**: index.html の StudyPanel（Level 2 / verse-flow）の中に属する。

---

### StudyPanel（index.html の底部パネル / #bottom-depth-panel）

**何をする場所か**:
- 語の意味を **理解する**（語義・読書メモ・Phrase Reading）
- 語の形を **分析する**（形態素グリッド）
- 近い語と **比較する**（セマンティックレイヤー）
- search-tool / morph-search / syntax-search へ **委譲する**（「さらに調べる」）

**4階層構造** (コード `5258`):
```
Level 1（常時表示）: 語義 + 読書メモ（散文）+ 使用傾向
Level 2（折りたたみ）: 単語を詳しく調べる（形態素 / 語源 / 辞書全文）
Level 3（折りたたみ）: 意味が近い語と比べる（セマンティック）
さらに調べる: 外部検索ツールへのリンク
```

**位置**: index.html に内包される。独立した画面ではない。

---

### Concept Insight（search-tool.html 内の知識カード）

**何をする場所か**:
- 検索語に対応するテーマを **知る**（概念名・関連ギリシャ語・関連テーマ）
- 関連テーマへの **発見の導線** を提供する
- パターン検索・構文分類への **橋渡し** をする

**位置**: search-tool.html の検索結果の上部。**検索結果とは別物** だが、同じ画面に同居している。

---

## 3. 責務分離マトリクス

| 責務 | search-tool | index.html | StudyPanel | 語順フロー | Concept Insight |
|------|-------------|------------|------------|-----------|-----------------|
| 探す（検索・絞り込み） | ◎ | ✗ | ✗ | ✗ | ✗ |
| 発見する（件数・分布） | ◎ | ✗ | ✗ | ✗ | ✗ |
| 理解する（テーマ・概念） | △(Insight) | ✗ | ◎ | △ | ◎ |
| 読む（本文精読） | ✗ | ◎ | ✗ | ○ | ✗ |

- search-tool.html が「読む」（Hit card 翻訳表示）と「分析（途中）」（語順プレビュー）を部分的に担っている
- 「分析（途中）」とは: 語順フローを表示するが、インタラクションが完結しない

## 4. ボタン監査

### 現在の状態

| ボタン | 動作 | ラベルと実態のずれ |
|--------|------|-------------------|
| 本文で見る | index.html?book=X&ch=N&verse=M へ遷移 | GOOD — 「見る」は適切 |
| 原文を見る | `hit-snippet-body` が展開。語順チップ（非クリック）を表示 | ✗ 「原文」=ギリシャ語テキストと誤解される。実際は日本語グロスの語順表示 |

---

### ボタン選択肢の比較評価

前提: 右ボタン（「原文を見る」）が表示するのは **「ギリシャ語順に並んだ日本語グロスのチップ列（非インタラクティブ）」** である。

| 選択肢 | 左ボタン | 右ボタン | 説明 |
|--------|---------|---------|------|
| A | 本文で読む | 語順フロー | 右が技術用語。初心者には意味不明 |
| B | 本文で読む | 語順で読む | 「読む」と言うがクリック不可。ミスリード |
| C | 本文で読む | 原語の流れを見る | 「原語」がギリシャ語を指すことが非自明。「流れ」は良い |
| D | 本文で読む | 語順を見る | 動詞「見る」が正確。「語順」は一定の専門性 |
| E | 本文で読む | （語順導線なし） | 最もクリーン。深掘りは index.html に任せる |

#### ペルソナ別評価

| 選択肢 | 初心者 | 一般信徒 | 牧師 | 神学生 |
|--------|--------|----------|------|--------|
| A「語順フロー」 | ✗ | △ | ○ | ○ |
| B「語順で読む」 | △ | △ | △ | △（動詞がミスマッチ） |
| C「原語の流れを見る」 | △ | △ | ○ | ○ |
| D「語順を見る」 | △ | ○ | ○ | ○ |
| E（右ボタンなし） | ○ | ○ | △（機能が消える） | △ |

#### 推奨

**左ボタン**: 「本文で読む」（「見る」→「読む」へ変更。行為が明確）

**右ボタン**: 以下の2案で設計方針が分かれる。

---

**方針1: 現在の inline flow を維持する場合**

→ **C「原語の流れを見る」** を推奨。

理由:
- 「原語」（ギリシャ語）を含むことで「専門的な表示」と暗示できる
- 「流れ」は index.html の「ことばの流れ」と語彙を統一できる
- 「見る」はインタラクション不要（非クリック）と整合する

制約: inline flow のチップを index.html と同様にクリック可能にするか、または非インタラクティブであることを明示する必要がある。

---

**方針2: inline flow を廃止する場合**

→ **E（右ボタンなし）** を推奨。

理由:
- 「本文で読む」→「節をタップ」→「ことばの流れ」（index.html）が正規ルート
- search-tool.html の inline flow は非インタラクティブで「途中で止まる体験」を作っている
- "本文で読む" 1ボタンだけにすることで、ルートが一本化される
- 初心者には何も失われない（flow は index.html で体験できる）
- 牧師・神学生は "本文で読む" から index.html に入れば同じ情報にアクセスできる

**設計上の注意**: 方針2を取る場合、Hit card の footer が「本文で読む」1ボタンだけになる。これは現在の `hit-open-btn-lg` 1つで成立する。

## 5. 画面責務の1行定義

| 画面 / コンポーネント | 責務の1行定義 |
|---------------------|--------------|
| **search-tool.html** | 「探す」——語・テーマ・番号で聖書全文を横断検索する場所 |
| **index.html（本文ページ）** | 「読む」——ギリシャ語と日本語訳を並べて本文を精読する場所 |
| **StudyPanel** | 「理解する」——単語の語義・形態・意味的位置を深掘りする場所 |
| **語順フロー（ことばの流れ）** | 「追う」——ギリシャ語順に沿って節のことばの流れを体感する場所 |
| **Concept Insight** | 「知る」——聖書のテーマと関連語の地図を一覧する場所 |
| **Concept Insight → 関連する概念** | 「広げる」——テーマから隣接テーマへ探索を展開する入口 |
| **さらに調べる（StudyPanel末尾）** | 「委譲する」——分析の深化を専門ツールへ引き渡す出口 |

## 6. 今後の設計原則（10項目）

### P1. search-tool は「探す」専業にする

検索ページの責務は「探す」と「発見する」に限定する。「読む」「理解する」「分析する」は別画面（index.html / StudyPanel）が担う。Hit card に翻訳を表示するのは「発見の補助」として許容するが、読書行為の代替にしてはならない。

### P2. 「本文で読む」が「読む」への唯一の正規ルートである

search-tool から読書体験への入口は「本文で読む」ボタン → index.html への遷移のみとする。search-tool 内で読書体験を完結させない。

### P3. 語順フロー（ことばの流れ）は index.html に属する

語順フローはチップのクリックで単語詳細（StudyPanel）へ繋がる「インタラクティブな読解 UI」である。search-tool 内の非インタラクティブな語順表示（hit-snippet-body）は「流れ」ではなく「構造プレビュー」であり、同一の UI 要素を使うべきでない。

### P4. 同一の UI 要素は同一の挙動を持つ

`.wlv-chip` が index.html ではクリック可能で search-tool.html ではクリック不可という状態は、ユーザーの期待を裏切る。コンポーネントは画面横断的に一貫した挙動を持つか、視覚的に区別する。

### P5. Concept Insight は「検索結果」ではなく「文脈の地図」である

Concept Insight は検索件数・ランキングに関与しない表示専用の知識カードである。検索結果リストとの視覚的な差別化を維持し、「概念を知る場所」として明示する。

### P6. 「探す」から「読む」へは1アクションで遷移できる

「本文で読む」1クリックで読書体験へ移行できること。中間ステップを増やさない。

### P7. 「読む」から「探す」へは明示的なアクションで遷移する

StudyPanel の「さらに調べる」セクションが search-tool.html への唯一の正規ルートである（concordURL）。この経路は「意図的な文脈切り替え」として設計する。読書中に意図せず検索ページへ飛ぶことを避ける。

### P8. 各画面は「どこへ進むか」を1方向にする

- search-tool → index.html（「本文で読む」）
- index.html（本文）→ StudyPanel（節タップ）
- StudyPanel → search-tool（「さらに調べる」）
- Concept Insight → search-tool（関連概念チップ = 同一画面内の再検索）

逆方向・中間段階のルートは混乱を生む。

### P9. 「探す」「読む」「理解する」の3段を段落として意識する

ユーザーがどの段階にいるかを常に意識する。「今あなたは探している」「今あなたは読んでいる」「今あなたは理解しようとしている」の文脈をUIが支持する。段落の混在はユーザーを迷子にする。

### P10. 「探す」の成果は「読む」への招待状である

search-tool の Hit card は「聖書のこの場所に、あなたが探していた語がある」という発見の報告である。それ自体が目的地ではなく、本文（index.html）への招待状である。「本文で読む」はその返答である。
