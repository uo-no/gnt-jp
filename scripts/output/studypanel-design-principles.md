# StudyPanel Design Principles（Phase 23.5 / 設計思想の正式確定）

- 確定日: 2026-07-07
- 位置づけ: **本書が StudyPanel の設計理念の正典（canonical）である。** 以後の情報設計・文書・実装判断は本書に整合させる。Phase 21–23・22.5 の監査レポートは本書に至る日付付き記録として原文を維持する
- 本フェーズでの変更: 文書のみ（docs/README.md 1 箇所 + 本書新設）。コード・Registry・テスト・UI 変更ゼロ

## 1. StudyPanel の正式な設計理念

> **StudyPanel は「ギリシャ語学習ツール」ではない。**
> **「ギリシャ語本文を通して聖書を理解するための読書支援ツール」である。**

- Greek Grammar は目的ではなく**根拠**である
- 利用者に見せたいものは、この順序に尽きる:

```
本文 → 意味 → まとまり → 流れ → 理解
```

- 文法は「**なぜそのように読めるか**」を支える説明層としてのみ登場する。読書の前に文法を置かない。文法から意味へ降りるのではなく、意味から（求められたときだけ）文法へ降りる
- 既存の二原則 — **静寂優先**（出せる内容がなければ何も出さない）と **ReadingFormatter 唯一の自然文生成源** — は本理念の実装上の柱として維持する

### 原則 1（Phase 23.6 追補）— 補助であって代行ではない

> **StudyPanel assists reading but never replaces reading.**
> StudyPanel は本文理解を**支援**するが、本文理解を**代行**しない。
> **すべての説明は、読者を本文へ戻すために存在する。**

これに伴う三命題を正式に確定する:

- **StudyPanel is not a commentary.** — 注解書ではない。本文の代わりに読む要約・解説を提供しない
- **StudyPanel is not a Greek grammar teacher.** — 文法教師ではない。文法体系の講義をしない
- **StudyPanel is a reading support layer** that helps users understand the biblical text **and return to the text itself.** — 理解の終点は常にパネルの外、本文の側にある

判定基準: どの説明も「読み終えたら本文のどこを見ればよいか」が判る形で終わること。パネル内で完結する説明（読まなくても済む説明）は、たとえ正確でも本理念への違反である。

## 2. 利用者モデル

- **日本語話者・ギリシャ語未習・神学校初学者レベル以下・聖書を理解したい読者**
- ギリシャ語を習得したい人ではない。Wallace を読む人ではない。文法を勉強したい人でもない
- 利用者がパネルに向ける問いは 4 つ（Phase 22.5 実証）: ①この単語は日本語で何か ②なぜその訳になるのか ③どこまでがひとまとまりか ④この形は前にも見たか — StudyPanel の成否は**この 4 問への回答品質**で測る
- 読書を続けた一部の読者は文法へ育つ。その人のための出口（Wallace 参照）は**常設・非強調の「卒業の扉」**として在る — 押して到達させるものではない

## 3. Wallace の位置づけ

> **Wallace = 教師ではない。読書メモを生成するための内部知識ベースである。**

- GGBB 1996 は分類の正確さ・用語・代表例の**典拠**であり、UI が教える教材ではない
- 「Wallace-based」の意味は「Wallace を教える」ではなく「**Wallace の統語論で読みの根拠を保証する**」である
- wallace_ref・description・alternatives は段階の最後（意味 → 例 → 名前 → 説明 → 別解 → 原典）でのみ現れる

## 4. Syntax Engine（242 型）の役割

> **242 の Syntax Types は UI の主役ではない。初心者の問いに、本文の言葉で、正確に答えるための裏方である。**

- エンジンの成果物（型判定・シグナル・確度・代表例・関連）は Learning Object — 正式には **Reading-support Projection（読書支援射影）** — を通してのみ表示層に届く。射影は文章を持たない（素材の運搬者であり文章の作者ではない）
- 情報の**地図**は Clause → Phrase → Word（Phase 23）で整理し、読者の**道筋**は Word（質問）→ Phrase（答えの単位）→ Clause（文脈の一言）（Phase 22.5）とする — 両者は矛盾せず役割が違う
- 型名（label_ja）は回答の冒頭に置かない。パターンに名前がつくのは再会と気づきの後（学習プロセス Step 8）

## 5. Reading Note の使命

> **Grammar Explanation ではない。「この箇所が何を言っているか」を本文の言葉で渡す 1 文である。**

- 文法用語ゼロ・1 文・読書の流れを止めない（現行テンプレの文体を維持）
- カテゴリの記述（「目的が示されています」）ではなく、**目の前のテキストの内容**（「『滅びないため』という目的」）を運ぶ — 型の説明文ではなくこの箇所の文
- 結論のみを言う。観察指示・根拠・確度の留保は Reading Note に混ぜない
- **結論は本文へ戻す形で言う**（原則 1）。説明で完結させず、本文の言葉を引いて「だからこの語順・この形で書かれている」へ読者の目を返す
- 文法名称は説明の**後**に置く（Phase 23.6 で順序を確定）: 本文の観察 → 意味 → 必要なら文法名称。
  - NG: 「これは主語的属格です。」
  - OK: 「『神の愛』は『神が愛する愛』として読むことができます。文法では主語的属格と呼ばれます。」

## 6. Observation の使命

> **文法訓練ではない。理解を助けるための「見る場所の指さし」である。**

- 登場条件: 意味を受け取った**後**、かつ**2 回目以降の遭遇時または読者が求めたとき**のみ（初回の遭遇では出さない）
- 内容: 「何を見るか」1 点を主とし、「どう考えるか」は解釈が割れる対（Wallace 自身が文脈依存とした場合）に限って判別基準 1 文を添える
- 目的は規則の暗記ではなく、グロス依存からの自然な卒業（理解 → 気づき → 一般化の順序）

## 7. StudyPanel が教えるもの

- **本文の意味** — この語・このかたまりが、この箇所で何を言っているか
- **まとまり** — どこからどこまでをひとつに読むか（かたまりの範囲と組み立て）
- **流れ** — καί・δέ・γάρ・ἵνα が話をどう進めるか
- **読める理由** — 求められたときに限り、なぜそう読めるかの根拠（術語なし → 名前 → 説明の段階で）
- **再会** — 前にも出会った形であること、聖書内の似た箇所

## 8. StudyPanel が教えないもの

- **ギリシャ語文法の体系**（格変化表の暗記・時制体系の講義 — 文法書の仕事）
- **Wallace の分類学それ自体**（242 型の一覧・分類名の網羅は UI の目的ではない）
- **解釈の断定**（解釈が割れる箇所では「決めなくても読み進められる」と言う — 不確実性は不安ではなく読書の常態）
- **翻訳の代替**（訳文はすでにある。パネルは訳と原文の間の「なぜ」を埋める）
- **専門用語の初期表示**（形態術語・型名・確度数値・marker を初期層に置かない — 既存原則の再確認）

## 10. 総合判定

**設計思想レベルでは、本プロジェクトは当初から一貫して「読書支援」を志向しており、「文法を教えるツール」への漂流は文書・コメントのいずれにも実質的に存在しない**（要修正は README の語順 1 箇所のみ）。

一方で Phase 21–23・22.5 が示したとおり、**実装の現状は思想を裏切っている** — 理念が求める「意味 → まとまり → 流れ」の層が空洞で、参照ツール的な内容（文法表・英語辞書）が前面にある。すなわち本プロジェクトの課題は「思想の修正」ではなく「**思想どおりに接続すること**」である。

本書をもって設計理念を確定する。以後の改善（Phase 24 以降）は、①本書 §1 の順序（本文→意味→まとまり→流れ→理解）、②§2 の 4 つの問い、③§5–6 の使命定義、④Phase 22.5 Audit 10 の優先順位（Critical: 即答・かたまり・重複解消）に照らして設計・評価すること。
