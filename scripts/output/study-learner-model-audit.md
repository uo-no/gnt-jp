# Learner Model Audit（Phase 22.5 / 学習者モデル監査）

- 監査日: 2026-07-07
- 想定読者: **日本語話者・ギリシャ語をほとんど知らない初学者**。神学生ではない。Wallace を読まない。文法を勉強したいのでもない。**目的は新約聖書を読めるようになること**
- 前提: Wallace-based Greek Syntax Engine（242 型）は裏側のエンジン。StudyPanel は「Wallace を教える UI」ではなく「聖書を読む UI」
- 方法: Phase 21–23 の事実（表示内容・データ構造）は参照するが、**結論は前提にせず初心者の認知からゼロベースで再評価**する
- 本監査での変更: コード・UI・Registry すべてゼロ（情報設計監査のみ）

---

## Audit 1 — 現在の StudyPanel は実際には誰のための UI か

**判定: 「文法を既習した中級者〜神学生」の参照ツールである。初心者向け要素は 2 つしかない。**

根拠（現在の表示内容の棚卸し）:

| 表示要素 | 前提としている読者 |
|---|---|
| 見出し語 + 日本語グロス | 初心者 ✓（唯一、問いに直に答えている） |
| 読書メモ 1 文（「ここでは、目的が示されています」型） | 初心者向けの文体だが、内容は後述のとおり空（Audit 4） |
| 形態グリッド（アオリスト・中動態・与格・大完了…） | **文法既習者**。これらの術語は初学者には未知語であり、日本語で書かれていても外国語に等しい |
| Abbott-Smith 辞書**全文** | **英語が読める神学生・研究者**。「日本語話者の初学者」という前提と正面から矛盾する |
| 語源合成・セマンティックレイヤー（意味領域・クラスタ） | 語彙研究をする人 |
| 「同じ語形を探す」「同じ構文を探す」リンク | 検索ツールを使いこなす中級者以上 |

つまり現在のパネルは「グロスの即答（初心者）」の直後に「文法表と英語辞書（神学生）」へ跳躍しており、**中間の読者（読めるようになりたい初心者）に語りかける層がない**。設計思想（静寂・用語を出さない）は初心者向けだが、実際に置かれている情報は参照ツールのそれである。

## Audit 2 — 日本語話者の理解が進むべき認知的順序

初学者は統語を上から降りて読まない。既知言語（日本語）を足場にした**意味先行・ボトムアップ**で進む:

```
Step A. 日本語訳で意味をつかむ（意味の仮説を先に持つ — 本アプリの並列表示が既に担う）
Step B. ギリシャ語本文と訳語の対応付け（「この語がこの意味か」— 読解フローのグロスが担う）
Step C. かたまりの認識（冠詞+名詞、前置詞+名詞が「一組」だと気づく — 意味的チャンキング）
Step D. 流れの把握（καί・δέ・γάρ・ἵνα が話をどう進めるか — 接続の感覚）
Step E. 再会と蓄積（同じ語・同じ形に何度も出会い、パターンが定着する）
Step F. （ずっと後で）文法体系（格・時制の表、用法の名前）
```

重要なのは、**節構造の分析（これは条件文だ）は初学者の理解プロセスに登場しない**ことである。節の性質は D の「接続の感覚」（もし〜なら、と話が進む）として体感され、「第3類条件文」という分析は F の段階まで不要。また C の「かたまり」は統語論の句（NP/PP）とほぼ一致するが、学習者にとっては**文法単位ではなく意味の単位**（「神の・ことば」でひとつ）として現れる。

## Audit 3 — Phase 23 の「Clause → Phrase → Word」は初心者に自然か

**判定: 情報の整理としては正しいが、学習の道筋としては不採用。初心者の道筋は逆向きである。**

- Phase 23 の階層は**分析者の下降**（構文解析の順序）である。初学者はそう読まない: 彼らが実際に起こす行動は「**わからない単語をタップする**」— 入口は常に Word である
- しかし Word の質問に Word 粒度だけで答えると失敗する（「θεοῦ は属格」は答えになっていない）。**答えの自然な単位はかたまり（Phrase）**である: 「τοῦ θεοῦ は直前の λόγος と一組で『神の・ことば』」
- Clause は答えではなく**周辺視野**（「この文は ἵνα から目的の話に入っている」）として最後に効く

### 提案する二重の導線（教育設計のみ）

| 入口 | 導線 | 用途 |
|---|---|---|
| 語をタップ（最頻の行動） | **Word（質問）→ Phrase（答え）→ Clause（文脈の一言）** | 読書中のつまずき解消 — 主経路 |
| 節を開く（節パネル） | **Clause（骨格）→ Phrase（かたまり）→ Word（詳細）** | 節全体を見渡したいとき — 副経路（Phase 23 の順序はこちらでのみ正しい） |

両経路の**要は Phrase 層**であり、この点で Phase 23 の中心的発見（Phrase 空洞が最大の欠落）はゼロベースでも再確認された。修正されるのは「主たる読み筋は Clause 起点ではなく Word 起点で、Phrase は降りる先ではなく**答えを返す単位**」という向きである。

## Audit 4 — Reading Note は初心者の最初の 1 文として適切か

現行例: 「ここでは、その目的として何が意図されているかが示されています。」

| 観点 | 評価 |
|---|---|
| 長さ | ✓ 適切（1 文・短い） |
| 専門用語 | ✓ ゼロ（設計原則が機能している） |
| 認知負荷 | ✓ 低い — しかし**効用も低い**。負荷ゼロで学びゼロの文になっている |
| 情報量 | **✗ 致命的に抽象的**。「目的が示されています」は情報の種類を予告するだけで、**肝心の目的の中身を言わない**。本文のどの言葉の話かも示さない |

さらに 2 つの構造問題が初心者に直撃する:

1. **同じ文が節パネルと単語詳細で 2 回出る**（Phase 23 の重複）。初心者は「押しても増えない」ことを 2〜3 回で学習し、パネルを開かなくなる — 学習性の無関心
2. 確度文「複数の解釈が考えられます」が、助けなしに**不安だけを足す**（どれとどれの解釈か言わないまま）

**あるべき姿（方針のみ）**: 読書メモは「型の説明文」ではなく「**この箇所の言葉を使った文**」であるべき。素材は既にある — 語のグロス（jaWord）・かたまりの範囲（ctx.phrases）・型の判定。「ἵνα 以下は『信じる者が滅びないため』という目的」のように、本文の内容を運ぶ 1 文が初心者の最初の文である。

## Audit 5 — Observation はいつ読むべきか

**判定: 「途中」— 意味を受け取った後、かつ求めたとき／2 回目以降の遭遇時。最初ではない。不要でもない。**

理由（習得順序の原則）:
- 言語の自然な習得は **理解（意味）→ 気づき（形式）→ 一般化（規則）** の順に進む。最初の遭遇で「ἵνα の後の動詞の形を見てください」と観察課題を出すのは、意味がまだ立っていない読者に演習を課すことになり、読書を止める
- 一方 Observation を捨てると「気づき」の段階が永久に来ず、読者はグロス依存から卒業できない。**2 回目以降の遭遇**（「この形、前にも見た」という感覚が生まれる瞬間）が観察指示の最適な投入時点である — 再会こそが観察の動機になる
- したがって設計上は: 初回 = 意味の即答のみ / 再会時 or 読者が自分で開いたとき = 観察 1 点（「〜に注目」）。「どう考えるか」まで添えるのは判別が割れる対（Phase 22 Audit 8 の規則）に限る

## Audit 6 — Wallace の情報はいつ見せるべきか（段階学習）

| 情報 | 段階 | 理由 |
|---|---|---|
| **example**（代表例） | **早め（第 2 段階）** — ただし「Wallace の例文」としてではなく「**聖書の中の似た箇所**」として | 初心者にとって最良の説明は規則ではなくもう 1 つの実例。JHN 3:16 で見た形を MAT 6:9 で再会させることはそのまま聖書を読む行為であり、読書 UI の思想と完全に両立する |
| **description**（用法解説） | 中盤 — パターンに名前がついた後、求めたとき | 名前（label_ja）を知る前に解説を読ませても留まらない。「これは『所有の属格』と呼ばれます」→「詳しく」の順 |
| **alternatives**（別解） | 遅く — 読者が「他の読み方はないのか」と問うたときのみ | 初心者に選択肢を並べるのは選択麻痺と不安の源。別解は問いが生まれてから初めて教育になる |
| **wallace_ref**（GGBB 頁） | 最後 — 卒業の扉として常設・非強調 | 本 UI の読者は Wallace を読まない前提。ただし読み続けた一部は文法へ育つ。その人のための出口は静かに在ればよい |

順序: **意味 → 例（聖書内の再会）→ 名前 → 説明 → 別解 → 原典**。

## Audit 7 — 初心者が最初に知りたいことと、パネルの回答状況

タップの瞬間に頭にある問いは、ほぼこの 4 つに尽きる:

| 問い | 現在の StudyPanel |
|---|---|
| Q1「この単語、日本語で何？」 | **✓ 答えている**（見出し語 + グロス。パネルで唯一機能している応答） |
| Q2「なぜこの訳になるの？」（語尾・格の働き） | ✗ 形態グリッド（術語の表）はあるが「答え」の形をしていない。242 型はまさにこの問いへの回答素材なのに未接続 |
| Q3「どこからどこまでが一まとまり？」 | ✗ 完全に不在（Phrase 空洞） |
| Q4「この形、前も見たけど同じ意味？」 | ✗ 不在。「最近見た単語」の保存機能はあるが、学習（再会の確認）には使われていない |

**4 問中 1 問にしか答えていない。** そして現在パネルが最も雄弁に語ること（節の抽象文・文法コード・英語辞書）は、初心者が尋ねていないことである。

## Audit 8 — 本文から Wallace までの理想的な学習プロセス

```
Step 1  日本語訳と並べて読む — 意味の仮説を持つ（既存機能）
Step 2  読解フローで語順のまま目を通す — 語と訳語の対応（既存機能）
Step 3  わからない語をタップ →「この語は『◯◯』。ここでは『…』と読める」— 意味の即答
Step 4  かたまりが示される —「τοῦ θεοῦ は前の λόγος と一組:『神の・ことば』」
Step 5  流れの一言 —「ἵνα からは『〜するために』と目的の話に入る」
Step 6  再会 —「この形は少し前にも出ました」/ 聖書内の似た箇所を 1 つ（example の読書的利用）
Step 7  求めたら理由 —「語尾が -ου なので『〜の』の形」（術語なしの文法）
Step 8  パターンに名前がつく —「この使い方は『所有の属格』と呼ばれます」（label_ja の初出）
Step 9  説明と代表例 — description / example を体系側から読み直す（Wallace 語彙の入口）
Step 10 Wallace へ — wallace_ref・alternatives・多層ビュー（研究の扉・完全に任意）
```

Step 1–6 が「読む」、Step 7–8 が「気づく」、Step 9–10 が「学ぶ」。**Wallace 到達は 10 段の最後**であり、押して到達させるものではなく、読み続けた人が引いて開くもの。現在の StudyPanel は Step 3 の半分（グロスのみ）と Step 9 の部品（英語辞書）だけがあり、**Step 4–8 の階段が丸ごとない**。

## Audit 9 — 「聖書を読む UI」として成功しているか（不足している考え方のみ）

成功していない。不足しているのは機能ではなく、次の**考え方**である:

1. **「聞かれたことに答える」** — パネルは語をタップされて開くのに、答える内容が節の講評になっている。質問の粒度（語）と回答の粒度（かたまり）と文脈の粒度（節）を区別する考え方が欠けている
2. **「本文の言葉で語る」** — 現在の文はすべてカテゴリの記述（「目的が示されています」）。読書の助けは常に目の前のテキストの言葉（「『滅びないため』という目的」）で語られるべき、という原則がない
3. **「再会が学習である」** — 読書は反復であり、同じ語形との 2 回目・5 回目の出会いこそ習得の瞬間。パネルは毎回を初対面として扱い、蓄積の概念を持たない
4. **「かたまりで読む」** — 語の列ではなく意味の単位で本文を見る、という読み方の教え自体が UI に存在しない
5. **「文法は卒業の扉」** — 文法情報（形態表・辞書・用法名）は常設の家具ではなく、読み続けた人が開く扉であるべき。現在は逆で、扉（文法）が居間に置かれ、居間（読みの助け）が物置にある
6. **「わからないままでよい」** — 解釈が割れる箇所（エンジンの Contextual 帯・競合候補）で、読書 UI は「決めなくても読み進められる」と言ってよい。不確実性を不安（「複数の解釈が…」）ではなく読書の常態として扱う考え方
7. **「読者の言語で」** — 英語辞書全文の常設は「日本語話者の初学者」という前提への違反。深層に退避させる段階設計が要る

## Audit 10 — 改善優先順位（初心者が学びやすくなる順）

### Critical（これがないと読書 UI として成立しない）
- **C1: Q2 への即答**（「なぜこの訳か」— この箇所の言葉を使った 1 文。242 型 + グロスが素材。Audit 4/7）
- **C2: かたまり（Phrase）の提示**（Q3。答えの単位・二重導線の要。Audit 3/7）
- **C3: 2 階層同一文の解消**（「押すと増える」の回復 — パネルへの信頼はここで決まる。Audit 4）

### High（読書の継続を生む）
- **H1: 流れの一言**（καί/δέ/γάρ/ἵνα — 語チップの手書き文言 CONJ_SIG は良質な原型。エンジンの conjunction/particle 判定と統合して拡張）
- **H2: 再会の設計**（2 回目以降の遭遇で Observation・似た箇所の提示。「最近見た単語」の教育的転用。Audit 5/7-Q4）
- **H3: 英語辞書全文の段階降格**（初心者層から深層へ。Audit 1/9-7）

### Medium（気づきから学びへの橋）
- **M1: パターン命名の段階規則**（label_ja をいつ初出させるか — Step 8 の設計）
- **M2: example の読書的利用**（「聖書内の似た箇所」として早期段階へ。Audit 6）
- **M3: 確度の語り直し**（「複数の解釈…」を不安でなく「決めずに読み進めてよい」へ — Phase 22 Audit 6 の指針と整合）
- **M4: 形態術語の敷居下げ**（アオリスト等に一言の添え書き — 表自体は Level 2 に畳まれており致命ではない）

### Low（育った読者のための出口）
- **L1: alternatives・多層ビュー・教育系列**（問いが生まれた読者専用）
- **L2: wallace_ref の常設・非強調**（卒業の扉）
- **L3: セマンティックレイヤー・語源の現状維持**（研究層向けとして既に適所）

## 結論

現在の StudyPanel は「聖書を読む UI」を志向した設計原則（静寂・用語排除）と、「神学生の参照ツール」の内容物（文法表・英語辞書・意味領域）が同居しており、**肝心の想定読者 — 読めるようになりたい日本語話者の初学者 — に語りかける層が空白**である。初心者の 4 つの問いのうち答えているのは 1 つ（グロス）だけであり、理想の 10 段の学習プロセスのうち Step 4–8 が存在しない。

Phase 23 の Clause→Phrase→Word はゼロベース再評価の結果、**情報の地図としては維持・学習の道筋としては逆転**（Word→Phrase→Clause を主経路に、Phrase を「答えの単位」に）という修正付きで引き継ぐ。Wallace エンジン（242 型）の役割は「教える内容」ではなく「**初心者の問いに、本文の言葉で、正確に答えるための裏方**」である — この位置づけの転換が、以後の全設計の出発点となるべきである。
