- 監査日: 2026-07-07

- 性質: **情報アーキテクチャ監査** — コード変更ゼロ・UI 変更ゼロ・UI 提案なし。教育アーキテクチャのみを評価する
- 対象: Wallace-based Greek Syntax Engine の全 242 型

## 1. 現状の情報フロー（Audit 1）

### 1.1 現状

```
Text（節トークン列）
  ↓
SyntaxAnalyzer.analyzeAll()          … 242 型 × 全トークンの候補列（label/hint/signals/conf/例/参照）
  ↓  ←←← ここで教育情報が全て脱落する（構造情報だけが下流へ渡る）
Phrase / Clause Analyzer             … 句・節の構造解析（syntaxResults は境界検出にのみ使用）
  ↓
discourse 判定（14 種）+ clause.type（5 種）
  ↓
ReadingFormatter._WALLACE_TEXT（47 文・うち 33 は到達不能）
  ↓
StudyPanel（読書メモ 1 文 + 確度 3 段階文）
```

### 1.2 情報が失われる 5 地点

| # | 地点 | 失われるもの | 性質 |
|---|---|---|---|
| L1 | SyntaxAnalyzer → ClauseAnalyzer | **候補列そのもの**（242 型の判定・rank・confidence・signals_matched） | 構造用途に縮約され、教育ペイロードが下流に存在しなくなる。**最大の損失点** |
| L2 | discourse 判定 | 節 1 つにつき 14 種 1 ラベルへ量子化 | トークン粒度 → 節粒度への不可逆縮約。単語をクリックしても節の 1 文しか語れない |
| L3 | ReadingFormatter.format() | hint（常に null）・confidence の中身（数値・根拠とも非開示） | 「marker を出さない」原則の過剰適用。根拠素材（signals label_ja）は設計上ここに届いてすらいない |
| L4 | `_CLAUSE_TYPE_TO_GLOSS_KEY` | genitive 22 + article 11 のテンプレ | 分岐は書かれているが到達経路が存在しない（死蔵） |
| L5 | StudyPanelAdapter | 5 カード構造（overview/clause/phrase/discourse/audit） | core 側に用意済みだが UI が未接続の孤児 API |

### 1.3 理想

```
Text → Parser → Syntax Engine → 【Learning Object】→ StudyPanel（Layer 1–4）
```

エンジンと表示の間に**教育情報を保持したまま通過させる中間表現**が 1 つもないことが、L1–L5 すべての共通原因である（§8）。

### 1.4 Study Flow の欠落（Audit 4）

現状の「分類 → 説明 → 読書メモ」は、実際には**分類が非表示・説明と読書メモが同一文**なので、学習サイクルとしては 1 ステップしかない。言語学習の観察サイクルに照らすと:

```
気づく（型名がある）→ 観察する（本文のどこを見るか）→ 根拠を確かめる（なぜそう言えるか）
→ 例と比べる（代表例との往復）→ 接続する（関連構文・Wallace へ）
```

のうち**「気づく」以外の 4 ステップに対応する情報が流れていない**。不足情報の列挙（UI 案ではない）: ①型名（label_ja）②観察指示（hint_ja / signals）③根拠（signals_matched + 加点）④代表例対応（example_greek/gloss）⑤競合解釈（第 2 候補・Δconfidence）⑥関連構文（alternatives）⑦原典参照（wallace_ref）⑧確度の語り（実分布に整合した段階）。

## 3. Learning Layers（Audit 3）+ 表示区分（Audit 5）

情報を**読書からの距離**で 4 層に整理する。各層の割当と、毎回/条件付き/上級者の区分:

| Layer | 想定者 | 置くべき情報 | 表示区分 |
|---|---|---|---|
| **Layer 1 — 読書**（現 rn-prose） | 初心者〜全員 | Reading Note 1 文（意味の要約・文法用語なし）。素材: 型ごとのテンプレ（description_ja の読書文体変換） | **毎回**。ただし top confidence < 0.40（Contextual 12 型）は**出さない**（現行の静寂優先を維持）— 誤断定回避は教育原則でもある |
| **Layer 2 — 観察** | 初心者後期〜中級 | ①型名 label_ja ②観察指示 hint_ja ③発火シグナル signals_matched の label_ja | **条件付き**: top ≥0.40 で開示可。**競合時（Δconf < 0.15 or 第 2 候補 ≥0.55）は観察が最も価値を持つ**ため、この条件をレイヤー昇格のトリガーに使う |
| **Layer 3 — Wallace** | 中級〜上級 | description_ja 全文・wallace_ref・wallace_freq・example_greek/gloss（代表例との対応） | **要求時**（折りたたみ相当の情報設計）。dormant 3 型は「NT に例が存在しない」旨を明示 |
| **Layer 4 — 比較・高度** | 上級 | alternatives（対比学習）・第 2〜3 候補と Δconf・多層並走（この語は接続詞層では X、節層では Y）・コーパス頻度 | **上級者のみ**（experiencedReader 相当のフラグは既存）。Contextual 帯（0.25–0.35）の候補も**この層でのみ**「提案」として開示 |

フィールド別の区分一覧: 毎回 = Reading Note 文のみ。条件付き = label_ja・hint_ja・signals_matched（≥0.40）、example（Layer 3 到達時）。上級者のみ = confidence 帯の意味・第 2 候補・alternatives ネットワーク・wallace_ref・多層ビュー・Contextual 候補。**「毎回」を 1 情報に絞る**のが本アプリの読書思想（静寂優先）との整合点である。

## 4. Reading Note の役割（Audit 9-前半）

- **専任**: 「このテキストが何をしているか」を**結論として 1 文で**言う。解釈レベル・文法用語ゼロ・読書の流れを止めない
- 現行 47 テンプレの文体（「ここでは、〜されています。」）は役割に適合しており維持すべき
- **持つべきでないもの**: 観察指示（→ Observation）・根拠（→ Layer 2）・確度の数値・型名。現状は混在こそないが、**確度文（「複数の解釈が考えられます」）が Reading Note に同居**しており、これは結論と留保の混在。確度の語りは Layer 2 以降の情報として分離すべき（§7）

## 5. Observation の役割（Audit 8・9-後半）

**「何を見るか」を主、「どう考えるか」は判別が分かれる場面に限って従**とすべきである。

- 教育設計上の根拠: 観察指示（notice）は学習者自身の気づきを生むが、思考手順まで毎回与えると読書が演習になり、本アプリの思想（読書を止めない）に反する。一方、**Wallace 自身が「文脈で決まる」とした対**（主語的/目的語的属格・ἵνα 目的/ὥστε 結果・第 1 類/第 2 類条件文）では、「見るだけ」では判断できず、**判別基準 1 つ**（「動作名詞なら: 動作するのは誰かを考える」）が不可欠
- 判定規則として整理できる: **単独発火型 → 見る箇所のみ**（hint_ja + signals_matched）。**競合型（alternatives が発火中 or Δconf 小）→ 見る箇所 + 判別基準 1 文**。この規則は registry の既存データ（alternatives・confidence）だけで機械的に決められる
- 役割分担のまとめ: **Reading Note = 結論（何が起きているか）/ Observation = 過程（どこを見て、迷ったら何で分けるか）**。両者を同一文に混ぜない — 現状は Observation 自体が存在しないため混在は未発生だが、導入時にこの境界を仕様化しておくべき

## 6. Related の役割 — 関連構文ネットワーク（Audit 7）

### 6.1 現状の alternatives グラフの実測

242 ノード・211 ペア（相互リンクは 71 = 34%）・**カテゴリ横断エッジはわずか 3 本**（gen.absolute↔ptc.temporal / dat.measure↔gen.comparison / clause.relative↔pron.relative）・**連結成分 56 個**（最大 35 ノード）。つまり現在の alternatives は「同一カテゴリ内の混同対」に閉じた断片群であり、仕様例のような比較連鎖

```
Predicate Nominative → Colwell → Predicate Adjective → Apposition
```

は**現データでは表現できない**（pred_nom の隣は subject のみ、colwell の隣は article 内のみ）。

### 6.2 比較学習の観点からの構成 — 3 種のエッジ

| エッジ種 | 意味 | 供給源 | 学習効果 |
|---|---|---|---|
| **E1 混同対**（既存 alternatives） | 「間違えやすい相手」 | registry（238 型整備済み） | 弁別学習（どこが違うかを見る） |
| **E2 多層並走**（導出） | 「同じ語の別レイヤー」（ἐάν = clause 0.70 / conjunction 0.65 / particle 0.60） | **解析出力から実行時に導出**（同一トークンの他カテゴリ候補） | 層の概念の獲得 — 1 語が複数の文法レベルで同時に機能することを体感 |
| **E3 教育系列**（導出 + 少量の編集） | Wallace の叙述順に沿った学習の道筋（pred_nom → colwell → …） | wallace_ref の頁隣接（同一頁共有 16 グループが既にある）+ カテゴリ内 priority 順 | 体系学習 — 点の知識を GGBB の地図に載せる |

E1 は完備、E2 は追加データ不要（実行時導出）、E3 のみ頁隣接から半自動生成 + 検収が要る。**56 断片を 1 つの学習地図にするのは E3 の役目**であり、これが Related 設計の中心課題である。

## 7. Confidence の扱い（Audit 6 — 改善指針のみ・文言は書かない）

実測: 中央値 0.60・分布 0.25–0.90・現行 3 段階の閾値（≥0.95 / ≥0.80）では **239/242 型が常に最下段の文言**に落ちる。改善指針:

1. **段階の境界を実分布の設計帯に一致させる**: 0.80+（形式で確定する構文）/ 0.60–0.75（標準判定）/ 0.40–0.55（弱い判定）/ <0.40（Contextual = 提案）の 4 帯。この帯は Phase 20 Audit 4 で「シグナル数と硬さ」に一対一対応することが検証済みなので、**文言は確率ではなく「根拠の数」を語るべき**
2. **確度は単独値でなく「差」で語る**: 学習上重要なのは top の絶対値より **top と第 2 候補の距離**（Δ < 0.15 = 解釈が割れる箇所 = 学び時）。syntax-search に既にある競合判定ロジックと同じ基準を語りの側にも使う
3. **数値非表示の原則は維持**しつつ、Layer 4 でのみ「なぜこの確度か」を signals_matched の列挙で説明可能にする（素材は全型完備）
4. Contextual 帯（0.25–0.35 の 12 型）は「低い確信」ではなく「**Wallace 自身が文脈依存とした用法**」という別カテゴリの語りにする — 確度の一次元尺度に混ぜないことが教育的に正確

## 8. Learning Object の必要性（Audit 10）

**必要と判定する。** 評価（設計のみ・実装論なし）:

```
Syntax Result（候補列） ─┐
registry typeDef ────────┼→ Learning Object（型ごと×箇所ごとの教育ビュー）
コーパス統計 ────────────┘        ├ reading   （L1: 結論 1 文）
                                  ├ observation（L2: 見る箇所・判別基準・発火根拠）
                                  ├ related   （L4: E1 混同対・E2 多層・E3 系列）
                                  ├ wallace   （L3: 解説・頁・頻度）
                                  ├ example   （L3: 代表例対応）
                                  └ level/条件（表示区分の判定済みフラグ）
```

- **賛成理由**: (a) §1.2 の損失 5 地点はすべて「教育情報を運ぶ中間表現の不在」に還元される — 個別に配線すると L1〜L5 それぞれに ad hoc な結合が生まれる。(b) 表示区分の規則（§3・§5）を**一箇所で**判定でき、Layer 設計の変更が 1 点に閉じる。(c) エンジン（Phase 20 で凍結監査済み）と学習コンテンツの進化速度が異なる — 疎結合の界面が必要。(d) 読み取り専用の射影であり、エンジン・registry・既存 API を変更せず成立する（禁止事項と両立）。(e) StudyPanelAdapter という先例（core 側の表示用射影）が既にあり、思想的に一貫する
- **リスクと限定**: 中間層は「第 2 の registry」になって内容が二重管理される危険がある。**LO は文章を保持しない**（テンプレと registry フィールドへの参照 + 実行時値のみを束ねる）という制約を設計原則として明記すべき。ReadingFormatter を唯一の自然文生成源とする既存原則とも矛盾しない（LO は素材を運び、文は従来どおり Formatter が作る）

## 10. 推奨アーキテクチャ（総括）

```
Text
 ↓
Parser（形態）
 ↓
Wallace-based Greek Syntax Engine（242 型・凍結監査済み）
 ↓            ↓             ↓
候補列     多層並走      コーパス統計
 └────────────┴──────────────┘
        Learning Object（読み取り専用の教育射影・文章は持たない）
              │  表示区分規則: 毎回 / 条件付き（≥0.40・競合時）/ 上級者
 ┌────────────┼────────────────────────────┐
 L1 Reading   L2 Observation   L3 Wallace   L4 比較・多層・確度根拠
（結論 1 文） （見る箇所＋      （解説・頁・  （E1 混同対・E2 層・
  毎回・静寂    判別基準・        代表例）      E3 系列・Δconf）
  優先）        発火根拠）
 └──────────── StudyPanel（既存 4 階層と同型の深さ方向） ────────────┘
```

設計原則のまとめ: **①教育情報は縮約せずに運び、表示段階で絞る**（現状は運搬段階で捨てている）②**結論（Reading）と過程（Observation）を分離する** ③**確度は根拠の数として語り、競合は学び時として扱う** ④**Related は 3 種のエッジで「弁別・層・体系」の 3 方向の比較学習を担う** ⑤**LO は素材の運搬者であり文章の作者ではない**。
