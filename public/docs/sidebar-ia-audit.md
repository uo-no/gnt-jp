# Sidebar Information Architecture Audit
version: 2.0
category: Design Canon Audit
scope: search-tool.html — .search-panel

---

## 監査の構造

**IA-00 が Sidebar の単一責務（Canon）を定義する。**  
IA-01〜IA-06 はその Canon への準拠を、各観点から個別に検証する。

```
IA-00: Single Responsibility  ← Canon（以降すべての前提）
  │
  ├── IA-01: Thought Order Alignment
  ├── IA-02: Approach Coordinate Parity
  ├── IA-03: Semantic Label Accuracy
  ├── IA-04: Choice Structure
  ├── IA-05: Progressive Disclosure
  └── IA-06: History Context
```

> **Note**: Search Funnel（Discovery → Search → Refinement）の監査は本 Audit の対象外。  
> 全体 UX フロー監査（Search Funnel Audit）が担当する。

---

## IA-00: Single Responsibility（単一責任原則）

**Sidebar の Canon:**

> Sidebar は「検索の実行」と「検索の履歴」のみを担う。

この一文が、Sidebar に含まれるすべての要素の正当性を判断する基準となる。  
「検索の実行」または「検索の履歴」のいずれにも属さない要素は、Canon 違反として扱う。

IA-00 は監査項目であると同時に、以降の IA-01〜IA-06 を成立させる前提条件である。  
IA-00 が FAIL の場合、他の項目の評価は意味をなさない。

**検証の問い:**  
Sidebar 内の各要素について「これは検索の実行に属するか、検索の履歴に属するか」を即答できるか。

**PASS 条件**
- Sidebar のすべての要素が「検索の実行」または「検索の履歴」のいずれかに分類できる
- 「これはなぜここにあるか」を各要素について即答できる
- Sidebar を見た第三者が1文で「このパネルは○○のためにある」と表現できる

**FAIL 条件**
- Discovery 要素（概念チップ・テーマ語）が Sidebar に存在する
- Navigation 要素（書物一覧・設定へのリンク）が Sidebar に存在する
- Result Refinement 要素（検索後の絞り込み）が Sidebar に存在する
- 「これが何のためにここにあるか」を即答できない要素が存在する

**よくある違反例**
- Discovery Lobby のテーマ語を Sidebar に追加する（Discovery は右パネルの責務）
- 検索実行ボタンを Sidebar の Primary CTA として配置する（実行は入力欄が担う）

**改善の優先順位**: P0 — これが FAIL の場合、他のすべての監査は意味をなさない

## IA-01: Thought Order Alignment（思考順一致性）

**目的**  
Sidebar の要素が「何を探すか → どう探すか → 過去に探したもの」というユーザーの自然な思考順に並んでいるか。

**評価基準**

| 思考フェーズ | Sidebar での対応要素 | 位置 |
|------------|-------------------|------|
| 「何を探すか」| 検索入力欄 | 最上部 |
| 「どう探すか」| 全検索アプローチ（語根/フレーズ/近接語/品詞）| 入力欄の直下 |
| 「過去に探したもの」| 履歴 | 最下部 |

**PASS 条件**
- 各フェーズの要素が上から順に配置されている
- 「どう探すか」フェーズの要素が完全に露出している（折りたたみなし）
- 思考フェーズをまたいだ要素の混在がない

**FAIL 条件**
- 「どう探すか」の要素の一部が折りたたみ、または別グループに隠されている
- 履歴が検索アプローチより前に位置している
- 「何を探すか」と「どう探すか」を分離するグループ境界が存在しない

**よくある違反例**
- 一部の検索アプローチのみ常時表示し、残りを "Advanced" として折りたたむ
- 履歴を入力欄の直下に配置する（「どう探すか」をスキップさせる）

**改善の優先順位**: P0 — 思考順の逆転は全体の IA を破壊する

## IA-02: Approach Coordinate Parity（アプローチ同格性）

**目的**  
複数の検索アプローチ（語根・フレーズ・近接語・品詞）が、ユーザーに同格として知覚されるか。

**評価基準**  
1つのアプローチが他と異なる IA 上の扱い・配置・命名を受けていないこと。

**PASS 条件**
- 全アプローチが同一の構造的位置に存在する
- 全アプローチが同一の折りたたみ状態（すべて表示 or すべて折りたたみ）にある
- 1つのアプローチが「デフォルト正解」として構造的に優位に置かれていない

**FAIL 条件**
- 特定のアプローチが他の前に単独で露出している
- 一部のアプローチが "詳細"・"Advanced"・"上級" 等の階層ラベルの下に置かれている
- 選択状態の表現により、特定アプローチが「通常のアプローチ」として構造化されている

**よくある違反例**
- `語根で探す` を常時露出し、`フレーズ` `近接語` `品詞` を "詳細検索" 内に折りたたむ
- Default-on の検索モードを他のアプローチとは別の構造的位置に配置する

**改善の優先順位**: P0 — アプローチの非同格化は Feature Discovery を根本から阻害する

## IA-03: Semantic Label Accuracy（ラベルの意味的正確性）

**目的**  
Sidebar 内のすべてのラベルが、実際の IA 構造と一致した意味を持っているか。

**評価基準**  
ラベルが以下の条件を満たすこと:
- グループの「性質」を正しく表現している（難易度や重要度ではなく、機能の種類を示す）
- ユーザーが読んだ瞬間に「これは自分に関係あるか」を正しく判断できる
- 誤った階層・誤った排他性を示唆しない

**PASS 条件**
- 「詳細検索」「高度な検索」「Advanced」等の難易度示唆ラベルが存在しない
- グループラベルがある場合、「アプローチ」「検索方法」等の中立的な表現を使用している
- 各アプローチのラベルは動詞句または名詞句で統一されている

**FAIL 条件**
- 「詳細」「高度」「上級」を含むラベルが、同格アプローチのグループに使われている
- ラベルが「この機能は使わなくていい」という読み取りを許容する
- グループラベルと内容物の意味的関係が成立しない

**よくある違反例**
- `› 詳細検索` — フレーズ・近接語・品詞を "advanced" として格下げする
- `もっと詳しく` — 探索を「必要な人だけ進む」行為として演出する

**改善の優先順位**: P0 — ラベルはアーキテクチャそのもの。誤った名称は誤った IA を生む

## IA-04: Choice Structure（選択肢の構造）

**目的**  
検索アプローチの選択肢が、ユーザーに「1つの統一されたグループ」として構造化されているか。

IA が評価すべきは選択肢の「数」ではなく、選択肢の「構造」である。  
同格の選択肢が凝集しているか、分散しているか——この構造的配置がユーザーの理解を規定する。

**評価基準**

| 評価軸 | PASS | FAIL |
|--------|------|------|
| 凝集性 | 全アプローチが1コンテナに収まっている | アプローチが複数のコンテナに分散している |
| 対等性 | 同一階層に並んでいる | 一部が上位、一部が下位コンテナに属する |
| 境界明確性 | グループの開始と終了が明確 | グループ境界が他の要素と混在する |

**PASS 条件**
- 4つの検索アプローチ（語根・フレーズ・近接語・品詞）が1つのグループとして認識できる
- グループ内のすべてのアプローチが同一の構造的レベルに存在する
- グループの境界が入力欄・履歴と明確に分離している

**FAIL 条件**
- 検索アプローチが「常時表示グループ」と「折りたたみグループ」に分裂している
- 1つのアプローチだけが他と異なる構造的コンテナ（カード・disclosure 等）に属している
- ユーザーが「これらは同じグループだ」と知覚できない配置になっている

**よくある違反例**
- 語根検索が単独カードとして配置され、残り3つが別の disclosure コンテナに入る（= 2グループに分裂）
- 「語根 + 詳細検索」という構造が、ユーザーに「2つの選択肢（語根か、詳細か）」と読まれる

**改善の優先順位**: P0 — 構造の分裂は「検索アプローチが4種ある」という事実をユーザーから隠す

## IA-05: Progressive Disclosure（段階的開示）

**目的**  
Sidebar の情報構造が、ユーザーの操作の進行に応じて適切なタイミングで情報を開示するよう設計されているか。

Apple HIG の Progressive Disclosure は「初期状態で不要な情報を表示しない」ではなく、「**複雑さが必要になった時だけ現れる**」という情報構造の設計原則である。  
本項目は Visual Weight・Density ではなく、「どの情報がいつ現れるか」という IA 上の時系列設計を評価する。

**評価基準**

| タイミング | 開示すべき情報 | 開示すべきでない情報 |
|-----------|------------|------------------|
| 初期状態（入力前）| 検索入力欄、検索アプローチ一覧 | 検索結果、履歴の詳細 |
| 入力中 | 入力候補（サジェスト）| アプローチの詳細設定 |
| アプローチ選択時 | 選択したアプローチの設定オプション | 他アプローチの設定オプション |
| 検索実行後 | 結果（右パネル）、履歴（左）| 未使用アプローチの詳細 |

**Progressive Disclosure と False Hierarchy の区別:**

| 種類 | 例 | IA 上の意味 |
|------|-----|-----------|
| 正しい Progressive Disclosure | 品詞の詳細指定（品詞アプローチ選択後に現れる）| 複雑さが必要になった時だけ現れる |
| 誤用（False Hierarchy）| `› 詳細検索` に同格アプローチを隠す | 同格の選択肢を階層化して排除 |

disclosure 構造が存在する場合、それが「後で必要になる複雑さを遅延開示する」ためか、「同格の選択肢を格下げするため」かを判断する。前者は正しい Progressive Disclosure、後者は IA-02・IA-03 の違反に加え本項目の違反でもある。

**PASS 条件**
- 検索アプローチの一覧は初期状態で開示されている（アプローチ選択は「いつ」ではなく「どれ」の問題）
- disclosure が存在する場合、それはアプローチ選択後に生じる詳細設定オプションに対してのみ使用されている
- 情報の開示タイミングがユーザーの操作フローと一致している

**FAIL 条件**
- 同格の検索アプローチ（語根・フレーズ・近接語・品詞）の一部が disclosure で隠されている
- Progressive Disclosure の形式を取りながら、実際には同格の選択肢を選別・排除している
- 品詞詳細フィルタ等のオプションが、アプローチを選択する前から常時展開されている

**よくある違反例**
- `› 詳細検索` という disclosure がアプローチ間の段階性ではなく優劣を作り出している

**改善の優先順位**: P1 — False Hierarchy を正しい Progressive Disclosure に置き換えることで IA-02・IA-03 も連動改善する

## IA-06: History Context（履歴の文脈的適切性）

**目的**  
履歴が「過去への参照」として適切なタイミング・場所に存在するか。

**評価基準**

履歴の表示タイミングが以下のいずれかに該当すること:
- A. 常時表示（下部固定）— 過去の検索が常に参照可能
- B. フォーカス時表示 — 検索入力に集中した時だけ現れる
- C. 表示なし — 結果画面の query label として代替

**PASS 条件**
- 履歴が Discovery Lobby（未検索状態）に主張として表示されない
- 履歴が検索アプローチ（語根/フレーズ等）と同一グループに置かれていない
- 表示タイミングが「検索を思い出したい時」に一致している

**FAIL 条件**
- 履歴が Discovery Lobby の視覚的中心に常時表示される
- 履歴と検索アプローチが区別なく並んでいる
- フォーカス時に履歴が表示されるが、blur タイムアウトが短くチップをクリックする前に消える

**よくある違反例**
- ページロード時に履歴チップが Discovery テーマと並んで表示される
- `blur` のタイムアウトが短く、履歴チップをクリックする前に消える

**改善の優先順位**: P2 — 機能的に動いていれば許容できるが、IA-00 の Canon 純度のために改善する
