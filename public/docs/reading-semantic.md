前提: Phase 1 Morphology / Phase 2 Syntax / Phase 3 Particle / Phase 4 Lexicon は

## 0. 最重要の監査発見 — Semantic は「推論」ではなく「注釈の読み出し」

NT 全巻監査で、**bible_data トークンが出現ごとの Louw-Nida 注釈
（token.ln・token.domain）を既に持っている**ことを確認した（カバレッジ 93.3%）。

例: λόγος のトークン別 ln 分布 — 33.98「ことば」184 件 / 33.26「話」81 件 /
57.228「勘定」9 件 / 89.18「理由」6 件…

したがって本プロジェクトの Semantic Layer は、実行時に文脈から意味を
**推論する層ではなく、Macula の語義注釈を読み出して対応する日本語を
引く層**として設計できる。これにより:

- 決定的（同じトークンは常に同じ結果）で監査・回帰固定が可能
- 「解釈を決めない」原則（reading-policy）と両立する
  （選択の根拠は既存の学術注釈であり、本エンジンの判断ではない）

> **設計原則: Phase 5 は実行時の文脈推論を一切行わない。
> 読むのは (a) トークン自身の注釈（ln / domain）、
> (b) キュレーション済みの対応表、の 2 つだけ。**

### 監査の測定方法（再現可能）

- カテゴリ 1: lemma 連鎖パターンの全巻走査
  （διὰ τοῦτο 67 / καὶ ἐγένετο 91 / ἀμὴν λέγω 69 / εἰς τὸν αἰῶνα 53 /
  ἐν τῷ+不定詞 41 / μὴ γένοιτο 27 / κατʼ ἰδίαν 18 / διὰ παντός 16 /
  ἐπὶ τὸ αὐτό 10 / εἰς αἰῶνα 2）
- カテゴリ 2: PP 目的語の domain 分類
  （ἐπί+属格の目的語: 場所 79・身体 16 / ἐν+与格: 場所 179・時間 115・人 272…

## 2. 「安全」と「危険」の分離

### 構造・注釈だけで安全に改善できるもの（Phase 5 対象）

1. **慣用句**: lemma 連鎖の完全一致。解釈判断ゼロ
2. **前置詞×目的語 domain**: ἐπί+属格+場所ドメイン→「〜の上で」等。
   注釈が明確なサブセットに限定（「その他」ドメインは fallback）
3. **ln 注釈駆動の語義選択**: token.ln → キュレーション済み対応表 → gloss。
   対応表に無い (lemmaId, ln) は fallback

### 意味判断が必要なので Phase 5 では扱わないもの

| 対象 | 除外理由 | 行き先 |
|---|---|---|
| 神学用語・比喩の訳し分け | reading-policy 原則（解釈を決めない・神学を教えない）に抵触 | **恒久対象外**。5-C では神学要注意 lemma のブロックリストで機械的に保護する |
| 接続詞・談話語（καί/δέ/ὅτι） | 談話機能の判断。既存チップ signal（「さらに、と積み重なる」等）と責務重複 | Reading Policy 2.0（ロードマップ旧 Phase 8）へ分離 |
| コプラ節の述語主格（は/が） | 意味判断ではなく**統語**判断の残課題 | Semantic ではなく **Particle 3-B** として将来分離（Phase 3-A の除外理由と同じ基準で再評価） |
| 句・文レベルの自然化 | トークン単位の resolve() の責務を超える | Phase 6 Phrase Renderer / 発展機能 Sentence Renderer |

## 3. Phase 分割

### 処理順序（resolve チェーン）

```
resolve(token, context?)
    ├─ _resolveSemantic   [Phase 5 新設 — 最上位]
    │     5-A 慣用句 → 5-B 前置詞domain → 5-C 語義選択 の順に照合
    ├─ _resolveSyntax     [FROZEN 2-A/2-B]
    ├─ _resolveMorph      [FROZEN Phase 1]
    ├─ _resolveParticle   [FROZEN 3-A]
    └─ null（fallback）
```

semantic を最上位に置く理由: 慣用句は syntax より優先されるべきだから
（εἰς τὸν αἰῶνα は syntax だと「世へ」だが、慣用句「永遠に」が正しい）。
**semantic が null を返す限り既存チェーンと完全同一** — 凍結保証の仕組みは
Phase 2/3 と同じ（監査の morphMismatch 検査を semantic 込みに拡張する）。

### Phase 5-A: 慣用句・固定表現

| 項目 | 内容 |
|---|---|
| 目的 | 逐語では読めない固定表現に句全体の訳を与える |
| 責務 | lemma 連鎖テーブル（`_IDIOM_TABLE`）の完全一致照合のみ |
| source | `'semantic'` |
| ResolveContext 追加 | **なし**（既存 tokens / targetIdx で隣接 lemma を照合） |
| 表示設計 | パターンの**内容語 head**（διὰ τοῦτο なら οὗτος）に句全体訳「このゆえに」を付与。付随語（前置詞・冠詞）は fallback のまま |
| 想定件数 | 初期 10 パターン・394 箇所。テーブル追加方式で拡張 |
| リスク | **低**。冠詞介在などパターン境界の検出精度のみ → 完全一致に限定し、監査でパターン別サンプル検証 |

### Phase 5-B: 前置詞の意味分岐（domain 注釈駆動）

| 項目 | 内容 |
|---|---|
| 目的 | 2-B で保留した多義前置詞を、目的語の domain 注釈が確定できるサブセットだけ解消 |
| 責務 | `前置詞 lemma × 格 × 目的語 domain クラス` テーブルの照合のみ（例: ἐπί+属格+場所→「〜の上で」79 件 / ἐν+与格+場所→「〜の中で」179 件 / περί+対格+身体→「〜の周りに」） |
| source | `'semantic'` |
| ResolveContext 追加 | **なし**（domain は目的語トークン自身のフィールド。PP head 判定は既存 phrases） |
| 想定件数 | 保守サブセットで 300–600 |
| リスク | 中。Macula domain 注釈の品質に依存 → 「その他」「なし」ドメインは必ず fallback。domain コードの桁数不統一の正規化を共通ヘルパー化 |

### Phase 5-C: 語義選択（ln 注釈駆動）

| 項目 | 内容 |
|---|---|
| 目的 | 多義内容語のトークンに、その出現の ln 注釈に対応する gloss を表示する |
| 責務 | `(lemmaId, ln) → gloss` 対応表の照合のみ。対応表整備は build スクリプトの責務 |
| source | `'semantic'` |
| ResolveContext 追加 | **なし**（ln はトークン自身のフィールド） |
| データ | **新規成果物: ln-gloss 対応表**。lexicon-lite（非同期）には依存させず、reading-lexicon-data.js と同様に**同期の生成データ**として焼き込む（4-C で特定した非同期非決定性の回避） |
| 想定件数 | 対応表の整備量に比例。高頻度シフト lemma 上位 50 語で約 5,000–8,000 トークン |
| リスク | 中〜高。対応表の品質が出力品質に直結 → **lemma 単位で段階投入**し、各投入分を回帰ケース化。神学要注意 lemma（υἱός・σάρξ・κόσμος・πνεῦμα 等）はブロックリストで初期から除外 |

**5-C の前提タスク（先行して実施）:**

1. **gloss 候補のノイズ除去**: 4-C の AS パーサが漏らした不完全括弧断片
   （「〜に（前置詞」）・メタ語（「等位接続」「転換語句」「通常の意味で」
   「属格を伴う場合」）を build スクリプトのフィルタ強化で排除する
2. **(lemmaId, ln) → gloss 対応表の構築方法の確立**: AS の①センス順と
   ln 配列は自動対応しないため、高頻度シフト lemma から機械支援
   （ln 定義の英語グロス × AS 候補の照合）+ 人手確認で段階作成する

## 4. ResolveResult / 監査・回帰の拡張

- `ResolveResult.source = 'semantic'`（ResolveSource.SEMANTIC は定義済み・未使用）
- 将来フィールド `confidence` は 5-C の対応表に品質段階を持たせる場合のみ導入
  （5-A/5-B は完全一致照合なので不要）
- 監査: `re-phase2-audit.cjs` に semantic 集計を追加
  （byIdiom / byPrepDomain / byLnGloss + 凍結違反検査を semantic 込みに拡張:
  「semantic 不発トークンは既存チェーンと同一出力」を機械検証）
- 回帰: `re-phase5-regression.cjs` を段階（5-A/5-B/5-C）ごとに拡張し、
  各段階完了時に基準値を凍結

## 5. Phase 5 の完了条件

1. 5-A/5-B/5-C 各段階で NT 全巻監査・悪化ケース 0・凍結
2. semantic 不発時の既存チェーン完全一致（凍結違反 0）を機械検証
3. 神学要注意 lemma ブロックリストが回帰ケースで固定されている
