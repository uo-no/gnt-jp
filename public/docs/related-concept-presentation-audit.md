```
動作: related.slice(0, N) を表示。残りは「他+X件」テキストで示すが非展開
```

| 上級者UX | △ 後続Conceptは完全非表示 |

| 実装コスト | 極小 |
| 問題点 | 救いの十字架(9番目)が非表示になる |

---

### Method C — priority 付き表示（JSON スキーマ拡張）

## 4. Priority 候補抽出

問題のある2 Concept（信仰・救い）について、UIで折りたたむ場合の分類案を示す。

### 救い (salvation) — 推奨 primary 5件

問題: キリスト(7番目)・十字架(9番目)が後方にある。表示順変更が前提。

**推奨表示順（5件 primary）**:
```
[信仰, キリスト, 十字架, 復活, 恵み]
```

**変更後 secondary（9件）**:
```
[希望, 罪, 悔い改め, 福音, 贖い, 永遠の命, 和解, 死, 赦し]
```

**根拠**: 「救い」を検索した初心者が次に読みたいのは「誰によって（キリスト）」「どのように（十字架・復活）」「何によって（信仰・恵み）」の5軸。神学的詳細（贖い・和解・赦し）は secondary で十分。

---

### 信仰 (faith) — 推奨 primary 4件

**推奨 primary（4件）**:
```
[救い, 愛, 祈り, 希望]
```

根拠: 1コリント13:13「信仰・希望・愛は残る」の三位一体 + 信仰の実践（祈り）

**secondary（7件）**:
```
[義, 真理, 従順, 忍耐, 律法, 証し, 誘惑]
```

削除候補（DATA_CHANGE検討）: 律法・証し・誘惑の3件は信仰の次に読む概念としては逆方向になりやすい。削除しても義・従順・忍耐で神学的深掘りは十分。

## 5. 結論

### 問題スコープの確認

related件数が過負荷な Concept: **2件のみ（5%）**  
→ 信仰(11件)・救い(14件) が対象。その他 35 Concept（95%）はすべて ≤7件で許容範囲内。

### 推奨判定

| Concept帯 | 判定 | 理由 |
|-----------|------|------|
| ≤7件（35 Concept）| **KEEP** | 全件表示で初心者UXに問題なし |
| 11件：信仰 | **IMPROVE** | UI 折りたたみで対応可能 |
| 14件：救い | **DATA_CHANGE + IMPROVE** | JSON 表示順の変更 + UI 折りたたみが必要 |

### 推奨実装: Method D（折りたたみ）+ 救いの表示順変更

**Step 1（DATA_CHANGE）**: `salvation.related` の配列順を変更  
```
現在: [faith, grace, sin, resurrection, hope, repentance, christ, gospel, cross, ...]
推奨: [faith, christ, cross, resurrection, grace, hope, sin, repentance, gospel, ...]
```
→ キリスト・十字架を先頭5件以内に移動する

**Step 2（IMPROVE / search-tool.html のみ）**: `_renderConceptInsight` に折りたたみ実装  
```
related件数 <= 8: 全件表示（現状と同じ）
related件数 > 8: 先頭8件表示 + 「他+X件を表示」ボタン
```

**この組み合わせの効果**:
- 信仰: 8件表示 [救い, 愛, 祈り, 義, 希望, 真理, 従順, 忍耐] + "+3件"[律法, 証し, 誘惑]
- 救い（順序変更後）: 8件表示 [信仰, キリスト, 十字架, 復活, 恵み, 希望, 罪, 悔い改め] + "+6件"
- 35 Concept: 変化なし（全件表示継続）
