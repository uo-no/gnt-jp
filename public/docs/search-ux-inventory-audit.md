## 4. 情報優先順位監査

### 4-1. 現在の表示順（検索後）

```
1. 結果ヘッダー（クエリ + 件数）
2. Concept Insight カード       ← 概念が発火した時のみ
3. 書物別分布パネル（バーグラフ）← スクロールが発生する
4. 件数テキスト
5. Lemma Groups（折りたたみ）   ← 日本語検索のみ
6. Hit Cards（最初の3件 + 展開ボタン）
```

### 4-2. 問題点

**書物別分布が Hit Cards より前に来る**  
ユーザーが最初に見たいのは「節の内容」（和訳）であるはずだが、  
分布グラフがスクロールを生んでいる。特にモバイルで顕著。

**推奨する表示順（UX-2 への提案）:**

```
1. 結果ヘッダー（クエリ + 件数）
2. Concept Insight カード
3. Hit Cards（先頭 3〜5 件）   ← コンテンツを先に
4. 書物別分布パネル             ← 文脈情報は後
5. Lemma Groups 全体（または残り）
```

### 4-3. Concept Insight 内の優先順位

現在の順（2025-07 UX-1 適用後）:

```
「X」について
→ 関連するギリシャ語（語彙 → 出現数 → 辞書）
→ 関連する概念
→ さらに詳しく ▸（関連する検索 / 関連する文法）
```

**評価: 適切。** 読書の自然な流れ（日本語概念 → ギリシャ語 → 関連概念 → 深掘り）に合っている。

### 4-4. Hit Card 内の優先順位

現在（UX-1 適用後）:

```
節参照 + 検索タイプバッジ（控えめ）
  └ 和訳テキスト（主要コンテンツ）
  └ [原文を見る] ← 副アクション
  └ [本文で見る] ← 主アクション（塗りつぶしボタン）
  └ 原文展開（Flow チップ）← hidden
```

**評価: 良好。** 「読む」目的への最短経路（和訳 → 本文へ）が機能している。

## 5. 重複箇所の整理

### 5-1. Pattern 入口の二重化

**箇所A:** Empty State「テーマから探す」  
**箇所B:** Concept Insight「さらに詳しく」→「関連する検索」

どちらも同じ 4 件の Pattern チップを提供している。  
A は検索前・全概念向け、B は特定概念発火後の文脈的な入口。  
役割が異なるため **現状維持でよい**。ただし B のラベルを「この概念の文法パターン」に変えると区別が明確になる。

### 5-2. Lemma 内共起 vs 共起語パネル

**箇所A:** Lemma Groups 展開時の「特徴共起」チップ（lg-chip）  
**箇所B:** 近接検索時の共起語パネル（右パネル左カラム）

A は日本語検索 × グループ内、B は近接検索 × 全体。  
同時に表示されることはないので機能上の重複はないが、  
**情報の読み方が同じ「共起」**であるにもかかわらず UI が別形式なのでユーザーが混乱する可能性がある。

### 5-3. 形態論フィルタの二重化

**箇所A:** search-tool.html の品詞・語形フィルタ（7 カテゴリ × チップ）  
**箇所B:** morph-search.html の形態論フィルタ（より詳細）

A は検索結果の絞り込みツール、B は形態論分析の専用ページ。  
目的が異なるため重複ではなく **相補的**。

### 5-4. 研究ツール（RT）タブの重複（index.html）

StudyPanel には「研究ツールを開く」コンテナが埋め込まれており、  
内部に **語形検索タブ / 構文検索タブ / 意味検索タブ** がある。  
これは search-tool.html / morph-search.html の機能を StudyPanel 内に再実装したもの。

- 同じ検索が 2 か所（リンク遷移 vs 埋め込みパネル）から起動できる
- 埋め込みパネルは画面が狭く UX が劣る
- 実用度が低い可能性がある

→ **削除候補として挙げる（後述）**

### 5-5. 結果ヘッダーバッジ（dead code）

`#result-layer-badges` は HTML / JS で生成されているが、  
CSS に `display:none` が設定されており **実際にはユーザーに表示されない**。  
Hit Card の `.hit-badges-subtle` で同等情報が提供されている。

→ **削除候補**

## 6. Progressive Disclosure の分析

### 6-1. 現在の開示レベル

**常時表示（全ユーザーに見える）:**
- 検索入力 + ボタン
- 語根レイヤー（ON）
- フレーズレイヤー（OFF）← 専門機能なのに常時露出
- 近接レイヤー（OFF）← 専門機能なのに常時露出
- 品詞・語形フィルタ（OFF）← 専門機能なのに常時露出
- 各レイヤーの説明文
- Hit Card の和訳
- 本文で見るボタン

**クリック・タップで展開（第2層）:**
- 近接距離スライダー + プリセット（prox ON 後）
- 品詞チップ + 詳細カテゴリ（morph ON 後）
- フレーズ説明文（phrase ON 後）
- Hit Card の「原文を見る」→ Flow チップ
- Concept Insight「さらに詳しく」→ Pattern / Wallace
- Syntax Filter Panel（`<details>`）
- Lemma Groups の各グループ

**3層目（第2層展開後）:**
- 品詞選択後の格・数・性・時制・態・法チップ
- Abbott-Smith 辞書モーダル

### 6-2. 第1層にある「専門機能」の問題

フレーズ・近接・品詞の 3 レイヤーは常時表示だが、  
初心者に必要ではない。レイヤーが OFF 状態でも存在感があり、  
何を選ぶべきか分からないという混乱が生まれうる。

**UX-2 への提案: 詳細オプションへの格納**

```
[検索入力] + [検索する]
[語根で探す] ← デフォルト ON のまま表示

[詳細オプション ▸] ← 折りたたみ
  └ フレーズで探す
  └ 近くに出てくる語（+ 距離設定）
  └ 品詞・語形で絞る（+ 詳細フィルタ）
```

これにより初心者は「語根で探す + 検索する」だけの UI を体験できる。

### 6-3. 書物別分布のタイミング

現在: 常時表示（結果後）  
推奨: Hit Cards 後 or `<details>` 内に移動

理由: 分布は「傾向分析」であり、まず節内容（和訳）を読んでから参照するもの。  
スクロールを減らすことで Hit Card への到達が早くなる。

## 7. 削除候補

### 7-1. `#result-layer-badges`（結果ヘッダーバッジ）

**理由:** CSS で `display:none` が設定されており、現在ユーザーに一切表示されていない。  
Hit Card の `.hit-badges-subtle` で同等情報が提供されている。  
HTML と JS のコードだけが存在する dead code。  
**削除しても UX に影響はない。**

### 7-2. Research Tools（RT）埋め込みタブ（index.html）

**場所:** StudyPanel → 「研究ツールを開く」コンテナ内の 3 タブ  
**理由:**
- search-tool.html / morph-search.html の機能を StudyPanel 幅（280〜420px）に詰め込んでいる
- 同じ機能が "さらに調べる" セクションの外部リンク（より良い UX）として既に存在する
- タブ内での操作は画面が狭く、フィルタ操作が困難
- ユーザーが自然に辿り着かない深さにある

**代替:** "さらに調べる" の 3 リンク（この語の用例を見る / この語形を探す / 同じ構文を探す）で完全に代替可能。  
**注意:** index.html は今回の監査対象のサブセット。削除には別フェーズでの判断が必要。

### 7-3. `#syn-stub-toggle`（参考値を含む）— 削除より非表示化を推奨

**理由:** 「参考値（確信度が低い機械的判断）を含む」という概念が専門家以外には理解困難。  
利用頻度が Very Low と推定される。  
**ただし完全削除より、デフォルト非表示（`<details>` のさらに奥）が適切。**

### 7-4. 近接プリセットの ±3 ボタン

**理由:** ±5 が「標準」、±10 が「広い」として機能的に十分。  
±3 は極めてニッチな用途（連続する 3 語以内の共起）で、  
大半のユースケースでは ±5 で対応できる。  
スライダーで任意設定も可能なため、プリセット数を 2 に減らすことを検討できる。

## 9. 総合評価

### 良い点

1. **検索エンジン自体は高品質** — Lemma / Phrase / Proximity / Morph の 4 レイヤーは機能的に完成している
2. **Hit Card の情報順序（UX-1 適用後）** — 和訳 → 本文で見る の流れは「読む」目的に合致
3. **Concept Insight（UX-1 適用後）** — 「X」について + Progressive Disclosure が初心者 / 専門家の両方に機能する
4. **モバイル対応** — Bottom Drawer + FAB の設計はスマートフォン向きとして適切
5. **データ品質** — 37 Concept × 4 Pattern という知識グラフは充実している

### 課題

1. **レイヤー UI が初心者に重い** — 4 レイヤーが常時見えており、何を使えばよいか分からない
2. **書物別分布が Hit Cards を押し下げる** — 最も重要なコンテンツへの到達が遅い
3. **サジェストが専門家向き** — 表示される情報（ギリシャ語 + 品詞コード + 件数）が初心者には読みにくい
4. **Dead code が存在する** — `#result-layer-badges` は使われていない

### ひとことで言うと

> 機能は完成しているが、**玄関に全室の鍵束が置いてある設計**。  
> 初心者はどれを使えばよいか迷い、牧師・神学生だけが効率よく使える状態。  
> Progressive Disclosure を徹底すると「誰でも検索でき、牧師・神学生は深く学べる」に到達できる。
