# Reading Flow Gloss Policy（Phase 5-C3C 設計正典）

作成: 2026-07-17
位置づけ: LN キュレーション（reading-ln-curation.js）を ReadingEngine へ接続する
前の判定基準。docs/reading-semantic.md（Phase 5 設計）の下位規範であり、
curation の採否・分類はすべて本書に照らして行う。

---

## 1. 前提 — このアプリの flow gloss とは何か

> **このアプリは翻訳聖書を作るものではない。
> ギリシャ語の語順のまま、日本語話者が構造を追うための語順フロー表示である。**

flow gloss（チップ・StudyPanel の語単位日本語）の役割:

- 読者は**ギリシャ語の語順**で gloss を追う。gloss は訳文の部品ではなく、
  各語への「指さし」である
- したがって gloss は**語単位で自立して**意味が通らなければならない
  （前後の日本語が文を成すことに依存できない）
- 句・構文レベルの意味は Phrase 層（5-A 慣用句・Phrase Reading）が担う。
  語 gloss は語義だけを担う

### 新改訳2017 の位置づけ

新改訳2017（および聖書協会共同訳）は**翻訳置換の基準ではなく、
神学的定訳を保護する基準**として使う。

- 主要訳が当該箇所で定訳（霊・肉・義・恵み・信仰 等）を**維持している**なら、
  flow gloss も定訳から離れない（読者の既存語彙との接続点を壊さない）
- 主要訳が当該箇所で**別の語**を使っている（例: MAT 6:1 δικαιοσύνη →「善行」）
  なら、それは「この箇所は定訳保護の対象外」という証拠になる
- 主要訳の訳語を**目標**にはしない（語順フローの gloss は訳文の語と
  役割が違う。一致はあくまで安全性の裏付け）

## 2. flow gloss として採用可能な条件（F1〜F6）

curation で固定訳を置き換える gloss は、以下を**すべて**満たす。

| # | 条件 | 内容 |
|---|---|---|
| F1 | **語単位自立性** | gloss はフロー中で単独で読める 1〜4 語程度の語義。句の説明・文の断片は不可 |
| F2 | **注釈駆動** | 置換の根拠は token.ln（Macula 注釈）のみ。文脈からの推論・解釈判断は禁止 |
| F3 | **神学的定訳の保護** | 主要訳が当該用例群で定訳を維持している場合は置換しない（→ preserve） |
| F4 | **破綻優先** | 置換の第一動機は「固定訳がその箇所で日本語として不成立、または意味が逆転する」こと。単なる好みの改善は動機にならない |
| F5 | **全用例一致** | その (lemmaId, ln) の**NT 全用例**で置換 gloss が成立し、主要訳の判断も割れていないこと。1 例でも不成立・訳割れがあれば replace 不可 |
| F6 | **可逆性** | curation は reason 付きで記録され、削除（＝auto へ復帰）でいつでも撤回できる。Engine 側は不一致時 fallback |

## 3. 三分類 — replace / assist / preserve

| 分類 | 定義 | Engine（Phase 5-D）での扱い |
|---|---|---|
| **replace** | F1〜F6 をすべて満たす。固定訳のままでは当該箇所が読めない/誤解される | jaWord を置換して表示（source: 'semantic'） |
| **assist** | 語義情報として価値があるが、F5（全用例一致）または F3 で replace に届かない。固定訳でも読める | **jaWord は置換しない**。StudyPanel の補助情報（「この箇所の語義: ◯◯」等）として提示する。表示設計は Phase 5-D で別途 |
| **preserve** | 神学的定訳・読者の語彙アンカー。LN 上の語義差があっても固定訳を守る | curation に入れない（= auto のまま。エンジン置換なし）。**保護判断そのものを記録する場合は curation ではなく本書 §6 の保護台帳に記す** |

### 判定フロー（ゲート式・上から順に適用）

```
(lemmaId, ln) と置換候補 gloss について:

G1 神学保護ゲート
   主要訳（新改訳2017 / 聖書協会共同訳）が当該用例群で定訳を維持
   → preserve（終了）

G2 破綻ゲート
   固定訳を当該用例に置くと日本語不成立 or 意味逆転か?
   → No なら「好みの改善」→ assist 止まり（または見送り）

G3 注釈明確性ゲート
   その ln の全用例で語義が一貫しているか?（代表聖句の実査）
   → 混在あり → 見送り（curation に入れない）

G4 全用例・定訳一致ゲート（F5）
   全用例で置換 gloss が成立し、主要訳の判断も割れていないか?
   → 1 例でも割れ → assist
   → すべて一致 → replace
```

## 4. curation データへの反映（Phase 5-D 前提のスキーマ拡張・設計のみ）

curation エントリに `mode` を必須フィールドとして追加する:

```javascript
"grc:G4151": Object.freeze({
    "23.186": Object.freeze({
        gloss: "息",
        mode: "assist",          // "replace" | "assist"
        reason: "…（従来どおり必須）",
    }),
});
```

- `mode: "replace"` のみが Engine の jaWord 置換対象
- `mode: "assist"` は最終辞書（reading-ln-final-data.js）に**別区画**で出力し、
  表示層が補助情報として消費する（build スクリプトの拡張は Phase 5-D で実装）
- preserve は curation に**書かない**（不在 = auto のまま = 置換なし）

## 7. Phase 5-D Engine 接続条件

以下を**すべて**満たすまで Engine（resolve 経路）に接続しない:

1. **mode フィールドの実装**: curation 全エントリに mode 必須。
   build:ln-final が replace / assist を別区画で出力し、監査が mode 欠落を検出する
2. **接続方式**: semantic 層の第 3 照合（idiom → prepDomain → **lnGloss**）として
   追加方式で実装。既存凍結部は不変更。照合は token.ln **先頭コード**の完全一致のみ
   （複合 ln はスペース区切り先頭 — 5-C1 生成と同じ規則）
3. **replace のみ置換**: assist は jaWord に触れない（補助表示の設計は接続と分離して先に固める）
4. **全件事前レビュー**: replace エントリの NT 対象トークン**全件**（現在 110 件 −
   assist 降格分）の出力を監査スクリプトでダンプし、目視レビューを通過すること
   （件数が小さいうちは全件レビューが可能 — サンプリングにしない）
5. **劣化モード**: READING_LN_FINAL 未ロード時は層不活性（既存チェーンと同一）
6. **回帰・基準値**: re-phase5-regression に lnGloss ケースを追加し、
   PHASE5 基準値の更新は件数内訳と理由を明記。悪化ケース 0・全 regression PASS
7. **保護台帳の機械化**: §6 の preserve 対象が誤って curation に追加された場合に
   監査（re-phase5c3a-audit）が警告すること
